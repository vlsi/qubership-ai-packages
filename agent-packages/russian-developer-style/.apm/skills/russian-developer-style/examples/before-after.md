# Примеры «до / после»

Калибровочный набор. Используйте как материал для сверки, когда непонятно,
насколько глубоко переписывать LLM-черновик. Примеры оригинальные.

## Документация и README (8)

### Пример: введение к разделу

Было:
> В данном руководстве будет рассмотрено, каким образом пользователь может осуществлять
> настройку повторных попыток выполнения HTTP-запросов с использованием параметра
> `retry.max_attempts`.

Стало:
> Настройте повторы HTTP-запросов через `retry.max_attempts`. По умолчанию — три попытки.

Почему:

- сразу действие, без «в данном руководстве»;
- «осуществлять настройку» → «настройте»;
- цепочка родительных свёрнута.

### Пример: описание сервиса

Было:
> `OrderService` представляет собой компонент, который является ответственным за обработку
> заказов и осуществляет взаимодействие с платёжным шлюзом.

Стало:
> `OrderService` обрабатывает заказы и общается с платёжным шлюзом.

Почему:

- две связки «является» / «представляет собой» убраны;
- глагол вместо «осуществляет взаимодействие».

### Пример: инструкция установки

Было:
> Для того чтобы установить пакет, вам необходимо выполнить следующую команду в терминале вашей системы.

Стало:
> Установите пакет:
>
> ```sh
> npm install @example/sdk
> ```

Почему:

- инструкция начинается с глагола;
- команда вынесена в блок;
- «для того чтобы», «вашей системы» — шум.

### Пример: обзорный абзац с тройкой

Было:
> Платформа предоставляет быстрый, надёжный и масштабируемый способ хранения событий.

Стало:
> Платформа хранит события и держит до 200k событий/с на узел при p99 < 50 мс.

Почему:

- триплет прилагательных заменён на проверяемое утверждение;
- «предоставляет способ» — пустая обёртка.

### Пример: финальный «обобщающий» абзац

Было:
> Таким образом, использование описанного подхода позволяет повысить надёжность системы
> и улучшить качество обслуживания пользователей.

Стало:
> *(удалено)*

Почему:

- абзац повторяет введение и ничего не добавляет;
- LLM-почерк «таким образом» + общая фраза.

### Пример: ссылка-приглашение

Было:
> Чтобы узнать больше о настройке, нажмите [здесь](docs/config.md).

Стало:
> Подробнее — в [`docs/config.md`](docs/config.md).

Почему:

- ссылка называет назначение, а не действие;
- инструкция «нажмите» избыточна в Markdown-документе.

### Пример: heading-кавалькада

Было:

> ## Введение
>
> ## Описание Проблемы
>
> ## Решение
>
> ## Заключение

Стало:

> ## Что делает сервис
>
> ## Когда использовать
>
> ## Как настроить
>
> ## Ограничения

Почему:

- заголовки описывают содержание, а не каркас «introduction/conclusion»;
- стиль обычного предложения, без капитализации каждого слова.

### Пример: список «возможностей»

Было:
>
> - Высокая производительность.
> - Масштабируемость из коробки.
> - Простая интеграция.

Стало:
>
> - До 100k записей/с на узел.
> - Горизонтальное масштабирование без даунтайма.
> - Драйверы для Go, Java и Python.

Почему:

- маркетинговые шаблоны заменены проверяемыми фактами;
- читатель получает информацию, а не настроение.

## UI-строки (8)

### Пример: кнопка

Было:
> «Осуществить сохранение»

Стало:
> «Сохранить»

Почему:

- кнопка — глагол в повелительном наклонении, без вспомогательного существительного.

### Пример: подпись поля

Было:
> «Пожалуйста, введите имя сервиса»

Стало:
> «Имя сервиса»

Почему:

- подпись — существительное, не инструкция;
- «пожалуйста» в UI лишнее.

### Пример: пустое состояние

Было:
> «У вас нет элементов. К сожалению, мы ничего не нашли.»

Стало:
> «Здесь пока ничего нет. Добавьте первое правило, чтобы начать.»

Почему:

- «к сожалению» — лишняя эмоция;
- даём конкретный следующий шаг.

### Пример: подтверждение удаления

Было:
> «Вы действительно уверены в том, что хотите осуществить удаление данного проекта?»

Стало:
> «Удалить проект? Восстановить его будет нельзя.»

Почему:

- короткий вопрос вместо нагромождения;
- предупреждение о необратимости — отдельным предложением.

### Пример: тост об успехе

Было:
> «Ваши изменения были успешно сохранены!»

Стало:
> «Изменения сохранены»

Почему:

- «были», «успешно», «ваши» — лишние;
- восклицательный знак в UI не нужен;
- без точки в конце короткого сообщения.

### Пример: плейсхолдер

Было:
> «Введите сюда ваш email»

Стало:
> «`user@example.com`»

Почему:

- плейсхолдер — пример, а не инструкция;
- подпись поля и так говорит «введите email».

### Пример: статус-индикатор

Было:
> «Идёт процесс обработки вашего запроса, пожалуйста, ожидайте.»

Стало:
> «Обрабатываем запрос…»

Почему:

- многоточие сигнализирует процесс;
- «процесс обработки», «пожалуйста, ожидайте» — канцелярит.

### Пример: tooltip

Было:
> «Эта настройка является ответственной за определение того, как часто будут выполняться повторные попытки.»

Стало:
> «Сколько раз повторять запрос при ошибке.»

Почему:

- «является ответственной», «определение того, как» — обёртка;
- описание короче и понятнее.

## Ошибки и логи (8)

### Пример: общая ошибка сохранения

Было:
> «Произошла ошибка при выполнении операции».

Стало:
> «Не удалось сохранить отчёт `report.csv`. Проверьте права на запись и повторите попытку.»

Почему:

- называет объект;
- даёт следующий шаг.

### Пример: ошибка валидации порта

Было:
> «Введённое значение некорректно».

Стало:
> «Значение порта `99999` вне диапазона. Укажите число от 1 до 65535.»

Почему:

- указывает, что именно неправильно;
- сразу даёт правило.

### Пример: ошибка истёкшего токена

Было:
> «Ошибка авторизации».

Стало:
> «Срок действия токена истёк. Войдите снова, чтобы продолжить.»

Почему:

- разделяет аутентификацию и авторизацию (здесь именно срок токена);
- даёт действие.

### Пример: сетевая ошибка

Было:
> «Что-то пошло не так. Попробуйте позже.»

Стало:
> «Сервер недоступен. Проверьте подключение и повторите запрос. Если ошибка повторяется, попробуйте через минуту.»

Почему:

- «что-то пошло не так» — антипаттерн;
- два конкретных действия для разных случаев.

### Пример: 500-я ошибка с поддержкой

Было:
> «Ошибка 500. Обратитесь в службу поддержки.»

Стало:
> «Сервер не смог обработать запрос. Попробуйте через минуту. Если ошибка повторяется,
> передайте в поддержку идентификатор `req-3f9c2b1a`.»

Почему:

- даёт пользователю шанс справиться самому;
- идентификатор позволяет поддержке найти событие.

### Пример: предупреждение об устаревшем файле

Было:
> «Конфиг устарел».

Стало:
> «Файл `legacy.yaml` устарел. Прочитан со значениями по умолчанию. Перейдите на `config.yaml` до релиза v3.»

Почему:

- что устарело, что система сделала, что сделать пользователю.

### Пример: лог подключения к Kafka

Было:
> `Ошибка при подключении к серверу`

Стало:
> `ERROR Не удалось подключиться к Kafka request_id=3f9c2b1a broker=kafka-1.prod cause="connection refused"`

Почему:

- уровень, объект, идентификатор, причина — всё в строке;
- структурированные поля, не проза.

### Пример: лог обработки очереди

Было:
> «Возникла проблема с обработкой сообщения».

Стало:
> `WARN Сообщение повторно обработано через DLQ message_id=msg-42 queue=payments-dlq attempts=4`

Почему:

- WARN — обходимое событие;
- идентификаторы, через которые видно событие в графане.

## Комментарии в коде и docstring (6)

### Пример: пустой комментарий

Было:

```go
// Получаем пользователя по идентификатору
user, err := repo.GetUser(ctx, userID)
```

Стало:
> *(комментарий удалён)*

Почему:

- сигнатура уже говорит то же;
- удаление комментария делает диф чище.

### Пример: комментарий с причиной

Было:

```go
// Округление вниз.
amount := math.Floor(value * 100) / 100
```

Стало:

```go
// Округляем вниз: биллинг отбрасывает дробные центы.
amount := math.Floor(value * 100) / 100
```

Почему:

- объясняет «почему», а не «что».

### Пример: docstring функции

Было:

```python
def load_config(path):
    """Этот метод используется для того, чтобы загрузить конфиг."""
```

Стало:

```python
def load_config(path: str) -> Config:
    """Читает YAML из path и применяет переменные окружения с префиксом FOO_.

    Переменные окружения переопределяют значения из файла.

    Args:
        path: путь к файлу конфигурации.

    Returns:
        Заполненную структуру Config.

    Raises:
        ConfigPathEmptyError: если path пустой.
    """
```

Почему:

- первая фраза описывает назначение;
- параметры, возвращаемое, исключения — на своих местах.

### Пример: Javadoc с лишней оболочкой

Было:

```java
/**
 * Данный метод предоставляет возможность зарегистрировать обработчик.
 */
```

Стало:

```java
/**
 * Регистрирует обработчик метрики и возвращает идентификатор для отмены.
 *
 * Регистрация идемпотентна: повторный вызов с тем же name переопределяет
 * предыдущий обработчик.
 */
```

Почему:

- «данный», «предоставляет возможность» — обёртка;
- появилось знание о семантике (идемпотентность).

### Пример: TODO без контекста

Было:

```ts
// TODO: переделать
```

Стало:

```ts
// TODO(#4271): после миграции на Fiber убрать обёртку.
```

Почему:

- ссылка на тикет;
- понятно, когда удалять.

### Пример: комментарий-парафраз

Было:

```go
// Увеличиваем счётчик на 1
counter++
```

Стало:
> *(удалено)*

Почему:

- следующая строка кода сама говорит за себя.

## PR-описания и changelog (4)

### Пример: размытое описание PR

Было:

```markdown
В этом PR внесены изменения в соответствии с требованиями.
Подробности — в коммитах.
```

Стало:

```markdown
## Что
Перевели обработчик `/orders` с net/http на Fiber.

## Зачем
Под нагрузкой net/http показывал p99 > 800 мс. На Fiber бенчмарк дал p99 < 200 мс.

## Как проверять
1. `make bench-orders` — p99 < 250 мс.
2. `go test ./api/orders/...`.
```

Почему:

- три блока: что, зачем, как проверять;
- конкретные числа и команды.

### Пример: changelog-запись

Было:
>
> - Улучшения и исправления.

Стало:
>
> - Исправили утечку файловых дескрипторов в `worker-pool` при отмене контекста (#4271).

Почему:

- глагол + конкретное изменение + ссылка на PR.

### Пример: release note про производительность

Было:
>
> - Произвели рефакторинг резолвера ради повышения производительности.

Стало:
>
> - Резолвер кэширует DNS-ответы 30 секунд: холодный старт быстрее на нагрузках с большим количеством подключений.

Почему:

- описывает изменение в терминах того, что меняется для пользователя;
- «произвели рефакторинг» → конкретное действие.

### Пример: breaking change

Было:
>
> - Изменили API.

Стало:

> ### Breaking changes
>
> - Удалили публичный метод `OrderService.ProcessLegacy`. Используйте `OrderService.Process`
>   с флагом `Legacy: true` (см. PR #4271).

Почему:

- breaking change выделен в отдельный блок;
- миграция описана конкретно.

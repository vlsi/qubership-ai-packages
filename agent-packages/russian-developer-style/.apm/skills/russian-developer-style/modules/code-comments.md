# Комментарии, docstring, PR и ревью

Этот модуль — продолжение раздела 9 основного [`SKILL.md`](../SKILL.md).
Подгружайте его, когда правите комментарии в коде, docstring,
Javadoc/KDoc/TSDoc, описание PR, ответы на код-ревью, фрагменты README,
встроенные в код.

## Inline-комментарии

Назначение комментария — передать информацию, которой нет в коде.

Что комментировать:

- **Зачем.** Бизнес-причина, инвариант, требование совместимости. «Округляем вниз, потому что
  биллинг не принимает дробные значения».
- **Почему именно так.** Когда из нескольких допустимых реализаций выбрана неочевидная.
  «Используем `sync.Once`, а не флаг, чтобы пройти валидацию `go vet`».
- **Подводный камень.** «Не выносить в горутину — порядок вставок важен».
- **Ссылка на тикет, RFC или баг.** «Workaround для kernel-бага 5.15+, см. issue #4271».

Что не комментировать:

- Повторение сигнатуры функции.
- Перевод имени переменной на русский («`userId` — это идентификатор пользователя»).
- Очевидное действие следующей строки («увеличиваем счётчик на 1»).

Длина: одна-две короткие фразы. Если нужен абзац — это, скорее всего, docstring или вынос в документацию.

Примеры:

```go
// Округляем вниз: биллинг отбрасывает дробные центы.
amount := math.Floor(value * 100) / 100
```

```go
// Workaround для https://github.com/golang/go/issues/12345:
// до 1.21 ResponseWriter.Write мог терять данные при concurrent Flush.
mu.Lock()
defer mu.Unlock()
```

Антипример:

```go
// Получаем пользователя по идентификатору.
user, err := repo.GetUser(ctx, userID)
```

Сигнатура уже сообщает то же. Комментарий — мусор.

## Docstring и Javadoc/KDoc/TSDoc

Шаблон:

1. **Первая фраза.** Короткое назначение, одна строка, без подлежащего «Метод/Функция».
1. **Подробности.** Семантика, инвариант, ограничения, побочные эффекты.
1. **Параметры.** Что значит каждый аргумент, единицы, диапазоны, что считается допустимым.
1. **Возвращаемое значение.** Что значит результат, что значит `nil` / `error` / пустая коллекция.
1. **Ошибки.** Какие исключения / возвращаемые ошибки бывают и в каких условиях.
1. **Пример.** Один компактный пример, если поведение неочевидное.

Идентификаторы кода в docstring не переводим, не склоняем, не оборачиваем в кавычки.

Пример (Go):

```go
// LoadConfig читает YAML из path и применяет переменные окружения с префиксом FOO_.
//
// Переменные окружения переопределяют значения из файла. Path должен указывать
// на существующий файл; пустая строка возвращает ErrConfigPathEmpty.
//
// Возвращает заполненную структуру Config или ошибку валидации. Ошибки io
// оборачиваются через fmt.Errorf с префиксом "load config:".
func LoadConfig(path string) (*Config, error)
```

Пример (Java/Javadoc):

```java
/**
 * Регистрирует обработчик метрики и возвращает идентификатор для отмены.
 *
 * Регистрация идемпотентна: повторный вызов с тем же {@code name} переопределяет
 * предыдущий обработчик.
 *
 * @param name уникальное имя метрики; не пустое
 * @param handler коллбек, вызываемый на каждом scrape-проходе
 * @return идентификатор регистрации; передайте в {@link #unregister(String)}
 * @throws IllegalArgumentException если {@code name} пустое или содержит пробелы
 */
String register(String name, MetricHandler handler);
```

Пример (TypeScript/TSDoc):

```ts
/**
 * Подписывается на изменения настройки и возвращает функцию отписки.
 *
 * Колбэк вызывается синхронно при первом подписывании с текущим значением
 * и затем при каждом изменении.
 *
 * @param key — ключ настройки
 * @param callback — обработчик; ошибки внутри него поглощаются и пишутся в консоль
 * @returns функция, отменяющая подписку
 */
export function subscribe(key: string, callback: (value: unknown) => void): () => void;
```

Антипаттерны:

- «Этот метод используется для того, чтобы получить пользователя» — три слова, ноль информации сверх сигнатуры.
- «Параметры: `userId` — идентификатор пользователя» — повтор имени без значения.
- «Возвращает: результат операции» — какой именно? Что значит «успех» и что значит «неуспех»?

## Описание PR

Минимальный комплект — три коротких блока в одинаковом порядке:

1. **Что.** Один абзац или список конкретных изменений. Не «Внёс правки и улучшения»,
   а «Перешёл с net/http на Fiber в `api/orders`».
1. **Зачем.** Проблема или ограничение, заставившее изменить код. Не «улучшаем код»,
   а «net/http не справляется с keep-alive под нагрузкой больше 5k rps».
1. **Как проверять.** Команды, скриншоты, имена тестов. Ревьюер должен понять, что запустить,
   не открывая diff.

Дополнительные блоки по необходимости:

- **Breaking changes** — что сломается у потребителей и как мигрировать.
- **Связанные тикеты / PR.**
- **TODO в следующем PR.**

Пример:

```markdown
## Что

- Перевели обработчик `/orders` с `net/http` на Fiber.
- Переписали middleware `auth` под Fiber-контракт.
- Удалили устаревший `OrderService.ProcessLegacy`.

## Зачем

Под пиковой нагрузкой `net/http` упирался в latency 90p > 800 мс. Бенчмарк
на Fiber показал 90p < 200 мс на тех же данных.

## Как проверять

1. `make bench-orders` — должен показать 90p < 250 мс.
2. `go test ./api/orders/...` — должны пройти все тесты.
3. Запустить `make integration` — смоук-тесты на staging.

## Breaking

Удалён публичный метод `OrderService.ProcessLegacy`. Потребители: `billing-svc`.
Перевели в этом же релизе, см. PR #4271.
```

Антипаттерны:

- «В этом PR были внесены изменения в соответствии с требованиями» — ноль информации.
- Описание, повторяющее заголовок коммита и больше ничего.
- «Подробности в коммитах» — ревьюер не должен реконструировать смысл изменения по diff.

## Ответы на код-ревью

Будьте конкретны и коллегиальны. Сарказм и канцелярит — мимо.

**Согласие.** Коротко, без витиеватости:

- «Поправил, спасибо.»
- «Согласен, переписал.»
- «Да, упустил, исправил в `f3a1c2b`.»

**Несогласие.** С аргументом и (по возможности) с примером:

- «Здесь нельзя выносить в горутину — порядок вставок важен для тестов
  в `TestSequentialInsert`. Если уберём порядок, упадёт.»
- «Согласен, что выглядит избыточно, но `sync.Once` страхует от двойной инициализации,
  которую трудно отловить в тестах. Оставлю как есть.»

**Уточняющий вопрос.** Не «я подумаю», а конкретно что нужно:

- «А что мы здесь хотим — ретраить идемпотентно или нет? От этого зависит, нужен ли мне `idempotency-key`.»

Антипаттерны:

- «Принято к сведению» — после этого никто не знает, что вы будете делать.
- «Не считаю это проблемой» — без аргумента воспринимается как отказ.
- «Возможно, имеет смысл рассмотреть вариант, при котором…» — переходите к делу: «Можно так:».
- «Спасибо за внимательность!» — на каждом комментарии воспринимается как формальное расшаркивание.

## Фрагменты README и обучающих материалов в коде

Если README встраивается в код через doctest, README-fenced-blocks или сгенерированные примеры:

- Имена переменных в примере выбирайте говорящие, не `foo`/`bar`, кроме случая, где код
  принципиально не привязан к домену.
- Не используйте русские идентификаторы в примерах, если в реальном коде они английские.
- Не зашивайте версии, если они должны браться из `go.mod` / `package.json` / `pom.xml`.
- Команды для копирования — без лишних `$` и оформления, чтобы Markdown «copy» работал из коробки.

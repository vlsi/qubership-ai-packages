# Ошибки, варнинги, валидации и логи

Этот модуль — продолжение раздела 8 основного [`SKILL.md`](../SKILL.md).
Подгружайте его, когда пишете тексты ошибок и валидаций для пользователя,
тексты варнингов и информационных сообщений, форматируете лог-строки.

## Общая модель

Различайте две аудитории:

- **Пользовательские сообщения** — пользователь системы, не разработчик.
  Простыми словами, без жаргона, с конкретным следующим шагом.
- **Сообщения для разработчика и оператора** — техническая плотность,
  идентификаторы, точные коды.

Один и тот же сбой может породить и пользовательское сообщение, и лог-строку.
Это разные тексты, а не один с разной обёрткой.

## Шаблон ошибки

Три фрагмента в строгой последовательности:

1. **Что не получилось.** Глагол + объект. По возможности — с подлежащим, если оно неочевидно.
1. **Почему.** Если причина известна и помогает понять, что делать.
1. **Что делать.** Конкретное действие или указание на источник.

Сводный шаблон:

> Не удалось `<действие>` `<объект>`. `<Причина, если известна>`. `<Что сделать>`.

Не все три части обязательны. Если причина непонятна — не выдумывайте её. Если действие
очевидно («попробуйте позже») — лучше укажите технический шаг.

## Примеры пользовательских ошибок

- *Было:* «Произошла ошибка при выполнении операции».
  *Стало:* «Не удалось сохранить отчёт `report.csv`. Проверьте права на запись и повторите попытку».

- *Было:* «Введено некорректное значение».
  *Стало:* «Значение порта `99999` вне диапазона. Укажите число от 1 до 65535».

- *Было:* «Ошибка авторизации».
  *Стало:* «Срок действия токена истёк. Войдите снова, чтобы продолжить».

- *Было:* «Server returned error» / «Сервер вернул ошибку».
  *Стало:* «Сервер недоступен. Проверьте подключение и повторите запрос».

- *Было:* «Ошибка 500».
  *Стало:* «Сервер не смог обработать запрос. Попробуйте ещё раз через минуту. Если ошибка
  повторяется, обратитесь в поддержку с идентификатором `req-3f9c2b1a`».

## Шаблон варнинга

Варнинг описывает ситуацию, которую программа обходит сама, но о которой пользователь или оператор должен знать.

Шаблон:

> `<Что произошло>`. `<Что система сделала или сделает>`. `<Что можно сделать>`.

Примеры:

- «Файл конфигурации `legacy.yaml` устарел. Поля прочитаны со значениями по умолчанию.
  Перейдите на `config.yaml` до релиза v3.»
- «Параметр `retries` не задан. Использовано значение 3. Чтобы задать явно, добавьте
  `retry.max_attempts` в `application.yaml`.»

Варнинг — не ошибка. Не пишите «Внимание! Ошибка…» — это путает уровни.

## Шаблон валидации формы

Валидация ловит ввод на лету. У такого сообщения короткая жизнь, поэтому оно должно сразу подсказывать исправление.

- Формат: «`<Поле>`: `<что не так>`. `<Подсказка>`.»
- Пишите рядом с полем, не в общем тосте.

Примеры:

- «Имя сервиса: содержит пробелы. Используйте буквы, цифры и дефис.»
- «Email: неверный формат. Пример: `user@example.com`.»
- «Пароль: 6 символов из 8 минимум. Добавьте ещё 2.»
- «Дата начала позже даты окончания. Поменяйте даты местами.»

Антипаттерны:

- «Поле заполнено некорректно» — без подсказки, что именно не так.
- «Ошибка валидации» — повторяет идею «ошибка», не информирует.
- «Введите корректное значение» — корректное по какому критерию?

## Шаблон лог-сообщения для разработчика

В логах прозы не нужно. Нужны:

- глагол в коротком повелительном или нейтральном времени: «failed to write file»,
  «kafka connection lost», «retry scheduled»;
- идентификаторы (trace id, request id, имя сервиса, ключ записи) — как структурированные поля, а не как текст;
- значимые значения (код ошибки, имя файла, путь);
- единый язык в проекте: либо весь лог по-английски, либо весь по-русски, не вперемешку.

Русский в логах допустим, если это явно решено в проекте; обычно лог-строки оставляют
на английском, чтобы не путать инструменты анализа.

Структура (русский вариант):

```text
2025-05-24T14:03:11Z ERROR Не удалось подключиться к Kafka request_id=3f9c2b1a broker=kafka-1.prod cause="connection refused"
```

Что недопустимо в логах:

- Обращение на «вы».
- Извинения и эмоции: «упс», «к сожалению», «что-то пошло не так».
- Длинные предложения с придаточными.
- Подстановка чужих фраз без идентификаторов: «Не удалось обработать запрос» — без request_id такой лог бесполезен.

Стандарт уровней:

- `DEBUG` — детальная диагностика, отключаемая в проде.
- `INFO` — нейтральный факт: «Сервис запущен на порте 8080», «Зарегистрирован обработчик `*ProcessorImpl`».
- `WARN` — обходимая ситуация: «Использовано значение по умолчанию», «Превышено мягкое ограничение».
- `ERROR` — реальная ошибка, требующая внимания. Не должна повторяться рутинно (иначе она информационная).
- `FATAL` / `PANIC` — невосстановимое состояние, сервис останавливается.

Не повышайте серьёзность в тексте без повышения уровня. «Критическая ошибка» на уровне `WARN` сбивает оператора с толку.

## Сообщения для оператора

Бывают тексты, обращённые к оператору SaaS / DevOps / SRE: алерты, страницы статусов,
exit-сообщения CLI. Это промежуточный регистр: технически плотно, как лог, но в виде
связных предложений.

Шаблон:

> `<Что сломалось>`. `<Что сделать руками>`. `<Ссылка на runbook или дашборд>`.

Пример:

> «Очередь `payments-dlq` накапливает сообщения. Проверьте обработчик `payments-consumer`
> и runbook <https://runbook.example/payments-dlq>.»

## Поля и инварианты

Если в сообщении пользователь видит подставленные значения, проверьте, что не сломаются крайние случаи:

- пустая строка: «Поле `` пустое» — нечитаемо. Подставьте имя поля из метаданных, а не значение.
- очень длинная строка: обрежьте до разумной длины и добавьте `…`.
- значение содержит управляющие символы (`\n`, `\t`): экранируйте или замените на читаемую запись.
- идентификаторы, которые пользователь не должен видеть (внутренние UUID, traceback):
  держите в логе, в UI показывайте короткий код для обращения в поддержку.

## CLI-вывод

CLI занимает промежуточное место: пользователь — разработчик, но он не читает лог в графане.

- Stdout — результат, который потом можно пайпить.
- Stderr — диагностические сообщения и ошибки.
- Не вмешивайте логи в stdout.
- Финальные строки с тоном вежливости (`Done!`, `Готово!`, `Спасибо за использование!`) — не нужны.
- Прогресс — без лишней анимации, читайте окружение: если не TTY, печатайте одну строку прогресса на каждые N процентов.

Шаблон ошибки CLI:

```text
error: cannot read config file '/etc/foo.yaml': permission denied
hint: run as a user with read access, or set FOO_CONFIG to a readable path
```

Русский вариант (если решено локализовать CLI):

```text
ошибка: не удалось прочитать `/etc/foo.yaml`: нет прав
подсказка: запустите под пользователем с правами на чтение или задайте FOO_CONFIG
```

## Антипаттерны

- **Размытое сообщение без идентификатора.** «Произошла ошибка». Никакой пользы.
- **Сваленные причина и обходное решение в одно предложение.** «Не удалось подключиться,
  проверьте соединение и попробуйте ещё раз, а если не помогает, свяжитесь с поддержкой» —
  разбейте.
- **Тавтология.** «Ошибка валидации: значение не прошло валидацию.»
- **«Уважаемый пользователь».** Не нужно, особенно в логе.
- **Извинения.** «К сожалению, мы не смогли…» — не сожалейте, сообщите факт.
- **Локализация частично.** Половина строки по-русски, половина по-английски. Согласуйте язык на уровне продукта.
- **Безличный конструкт с потерей подлежащего.** «Было обнаружено, что…» — кем, чем? Скажите субъекта.

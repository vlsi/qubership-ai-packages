# Phase 1 Discovery: Sources for a Russian Technical-Prose LLM Skill

## 1. Executive summary

**No single proven, off-the-shelf Russian LLM skill exists** for the target use case (technical documentation, READMEs, API docs, code comments, UI strings, error messages, release notes, PR text). The public `anthropics/skills` repository ships skills for `docx`, `pdf`, `pptx`, `doc-coauthoring`, `skill-creator`, `mcp-builder`, etc., but contains no Russian editing/authoring skill. Community SKILL.md / Cursor rule packs aimed specifically at natural Russian technical prose are not visible in the searches performed; the Russian-language repos that surface for "SKILL.md" labels (e.g. `VKirill/yandex-direct-docs`) are product-doc scrapings, not editorial rules.

**Strongest source category: Russian UI / localization style guides + Russian editorial classics.** The Microsoft Russian Localization Style Guide (`rus-rus-StyleGuide.pdf`) is an authoritative, software-specific, terminology-rich document covering UI tone, capitalization, and translation rules. Контур.Гайды (`guides.kontur.ru`) is the rare publicly maintained corporate редполитика explicitly written for software interfaces and documentation. Together with Ozon Tech's public styleguide (`docs.ozon.ru/styleguide`, publicly launched on 3 March 2025 by Аня Салугина) and the Mozilla L10n general guide, they cover most UI/doc rules natively in software context. Editorial classics (Нора Галь, Чуковский) provide the canonical taxonomy of канцелярит — directly mappable to the user's failure-mode list (отглагольные существительные, пассив, цепочки родительных падежей, "является/осуществляет").

**Weakest / most polluted category: "Russian LLM prose analysis."** Habr is the only practical venue, and the relevant posts ("Ваш текст воняет GPT", "Вы не сможете это развидеть") are useful primary observations but are unbylined blog posts that summarise rather than reproduce primary research. They risk over-pivoting the skill toward AI-detector evasion, which is explicitly out of scope. Generic "редполитика" copycats and "Главред"-derived journalism rules are also polluted: useful syntactic minimums (стоп-слова, отглагольные существительные) sit next to marketing-oriented advice that does not transfer to API reference docs.

**Russian UI/localization: yes, strong sources exist** — Microsoft Russian Style Guide, Контур.Гайды (текст в интерфейсе), Ozon UX-редактура курс + публичный styleguide, Mozilla L10n general guide, WordPress.com Russian Style Guide, JetBrains Russian Language Pack (via Crowdin community localization).

**Russian developer-doc style: medium.** Ozon Tech's public styleguide is the strongest publicly available corporate dev-doc styleguide; Yandex maintains an internal styleguide but has not released it — confirmed in the yandex.ru/blog/x-plain Q&A: "Мы продолжаем думать над тем, в каком формате и объеме выкладывать информацию из гайда." Habr posts by Yandex/Ozon technical writers describe principles but are not formal rule sets. There is no Russian equivalent of Google Developer Docs Style Guide with deep public coverage.

**Phase 2 recommendation:** combine three pillars — (1) UI/localization rules from Microsoft + Контур + Ozon for tone, terminology, button/error wording; (2) editorial classics (Нора Галь + Розенталь/Мильчин + Bureau "Советы") for канцелярит rules and syntax; (3) a small mechanical-checks layer (yaspeller, LanguageTool ru rules, Типограф Муравьёва / Студии Лебедева) kept *outside* the LLM skill in a linter pipeline. Habr LLM-tells posts contribute the failure-mode list, not positive rules. Avoid Главред as a primary source — its anti-marketing-mode rules conflict with technical precision (it flags valid terminology and modal verbs that are mandatory in API docs).

## 3. Promising shortlist for Phase 2

**1. Microsoft Russian Localization Style Guide PDF (#1).** Single most concrete, software-specific Russian UI source. Defines tone, capitalization (русский lowercase в середине заголовка), правила обращения, форматы дат/чисел, glossary patterns. Most useful for **UI translation + terminology**. Converts well into a SKILL.md section "UI-строки, кнопки, сообщения об ошибках" plus a terminology table. Caveat: Microsoft-specific terminology must be filtered.

**2. Контур.Гайды Редполитика + Текст в интерфейсе (#3).** The only publicly maintained Russian corporate styleguide that explicitly addresses both software UI ("текст в интерфейсе") and editorial principles ("Понятные предложения", "Простые слова"), with the named tone "помощник и эксперт" and an explicit "робот vs Контур" persona dichotomy. Mostly useful for **authoring + UI translation**. Converts cleanly into a compact rule set — Контур already wrote it as rules with examples.

**3. Ozon Tech publicly released styleguide (#4) + UX-редактура курс (#5).** The most recent and most explicitly "tech-writers' rules in Russian" public source (published 3 March 2025 by Аня Салугина on Habr 883502). Covers "Написание документации", "Голос и тон", интерфейсные элементы, орфография. Best for **authoring + editing developer docs**. Highly convertible to SKILL.md because it is already structured by topic.

**4. Нора Галь "Слово живое и мёртвое" (#10).** Provides the canonical taxonomy that *exactly* matches the user's failure-mode list: "вытеснение глагола причастиями, деепричастиями и существительными (особенно отглагольными)", "нагромождение существительных в косвенных падежах", "из всех глагольных форм пристрастие к инфинитиву", "штампы вместо живого образа". Best for **editing-rewrite**. Converts to a 10–15 rule block in SKILL.md with before/after examples. Risk: literary register — needs filtering so the skill does not push API docs toward fiction.

**5. Mozilla L10n general style guide (#6).** Software-native principles (accuracy, fluency, consistency, do-not-translate handling, cultural references, trademark rules) at the right altitude for SKILL.md. Best for **UI translation**. Converts well because it is already in markdown rule format on GitHub. Useful as a structural template for the skill itself.

**6. Habr "Ваш текст воняет GPT" + companion posts (#22, #23) + Reinhart et al. PNAS 2025 (#24).** Not for positive rules but for the diagnostic checklist of LLM-tells. Best as **background research → failure-mode catalogue** at the top of SKILL.md ("чего не делать"). Reinhart's multidimensional Biber analysis (PNAS 122(8) e2422455122) supplies the academic foundation; Habr summaries adapt it to Russian-specific phenomena (причастные/деепричастные обороты, кальки типа "играет ключевую роль", "на сегодняшний день").

**7. yaspeller + LanguageTool Russian (#16, #17).** Mechanical layer (spelling, ё-fication, repeated words, common punctuation/grammar). Best for **linting-review**. Should NOT live in SKILL.md — belongs in CI. Important caveat: yaspeller was archived by its maintainer on 24 April 2024 (last release v10.0.1, 12 November 2023); still functional, but Phase 2 should either fork it or replace it with LanguageTool 6.6 (930 ru rules, 27 March 2025) as the primary engine.

**8. Типограф Муравьёва / Лебедева (#18, #19).** Typography — кавычки-ёлочки, неразрывные пробелы, длинное vs дефис, тонкий пробел в числах. Best for **linting-review**, runs post-LLM. The skill should know *not* to overdo "—" but offload mechanical typography to a typograf.

**9. Bureau Gorbunov "Советы", рубрика Текст (#14).** Steady stream of practical editorial mini-rules from the same tradition as Ильяхов but more pluralistic (rubric "Текст" curated by Максим Ильяхов; 4770 советов total as of search date). Best for **editing reference + examples**. Less convertible to a single SKILL.md block because it is a corpus, not a ruleset; useful as example bank.

**10. JetBrains Russian Language Pack via Crowdin (#25).** A peer-reviewed Russian terminology corpus for developer-tool UI strings (commit, merge, refactor, request). Best for **terminology** — mine for a glossary section in the skill.

## 4. Obvious rejects

- **"AI humanizer" / detector-evasion services (Undetectable AI, Isgen reformatters).** Explicitly out of scope; optimise for fooling detectors rather than reader experience, and produce inconsistent, often anti-precise output.
- **"Главред" as primary skill content.** Drives toward journalistic/marketing "инфостиль" that explicitly attacks valid technical patterns (modal verbs like "необходимо", "следует"; passive voice that is mandatory in API descriptions). Useful only as stop-word inspiration. Even Ильяхов himself warns that if you "отжать тексты до инфостиля — они перестанут выполнять свою функцию" (vsevolodustinov.ru notes from Ильяхов's course). Keep for inspiration in #2, reject as authoritative source.
- **Тинькофф-Журнал редполитика.** Authors label it "Больше не обновляется. 18+. Можно вдохновляться". Journalistic, not technical.
- **WordPress.com Russian Style Guide (#7).** Too thin and too generic; one paragraph on imperative vs infinitive; included in inventory only for completeness.
- **Russian-grammar-checker SaaS marketing pages (SEOMagnifier, Scribens, "FlowWrite", "Орфограммка" listicles).** Affiliate/SEO surfaces with no editorial authority; only LanguageTool and yaspeller are genuine engines.
- **Generic "редполитика для блога" templates (Texterra, 1PS, Unisender, Neaktor).** Copywriting/SEO orientation; conflict with technical precision.
- **Yandex Direct documentation packaged as `SKILL.md` (`VKirill/yandex-direct-docs`).** Product-doc scrape labelled "Agent Skill", not an editorial rule set. Misleadingly tagged.
- **Russian-grammar-for-learners content (FluentU, "Learn a language online").** Pedagogical for non-natives; no editorial signal.
- **Anonymous Telegram / Pikabu prompt snippets for "очеловечивание" LLM-русского.** No provenance, no editorial review.
- **Ozon/Wildberries seller-card SEO blogs (SelSup, MPAgency).** Appeared in searches for "Avito/Wildberries engineering style"; entirely about marketplace listings, irrelevant.

## 5. Gaps and questions for Phase 2

1. **Microsoft Russian PDF — date and currency.** Confirm publication year (the download link is stable but not dated on the listing page) and whether it aligns with the live Microsoft Writing Style Guide (English-first). Whether deprecated UI conventions are still inside.
2. **Yandex internal styleguide.** Repeated public statement at yandex.ru/blog/x-plain/474: "Мы продолжаем думать над тем, в каком формате и объеме выкладывать информацию из гайда." Phase 2 should mine secondary descriptions (Гипербатон talks, Habr posts by Андрей Поляков, Юлия Пивоварова, Татьяна Грачёва) as proxy.
3. **Ozon styleguide depth.** Does docs.ozon.ru/styleguide cover code-comment / Javadoc-style writing, or only end-user docs and UI? The Habr 883502 structure (Написание + Оформление) suggests UI/help-centric — verify before relying on it for code-comment rules.
4. **Канцелярит vs technical precision.** Нора Галь's rules push toward verbs and away from отглагольные существительные, but technical Russian needs nominalisations ("инициализация", "сериализация", "выделение памяти"). Phase 2 must list which Галь rules are unconditional and which need a "технический контекст" override.
5. **Tone selection ты/вы/безличный.** Контур, Ozon, and Mozilla each pick differently. Skill must take an explicit position per text type:
   - UI (button/menu) — императив, безличный ("Сохранить", "Удалить", не "Сохраните")
   - error messages — безличный + что произошло + что делать ("Не удалось…", не "Вы не смогли")
   - tutorials/READMEs — единое лицо (Контур: «мы»; Ozon: первое лицо)
   - API reference — безличный / инфинитив-описательный
   - PR/review replies — "вы" нейтральное + действие
   Phase 2 task: pick defaults and document deviations.
6. **Terminology automation.** Glossary (yaspeller dict + JetBrains corpus + Microsoft PDF terminology pulls) can be mostly automated. Question: lives in the SKILL.md, in `.yaspellerrc.json`, or in a project-level glossary file the skill *references*?
7. **Mechanical layer split.** Confirm thesis: typography (кавычки, неразрывные, тире), spelling, ё-fication, double spaces → linter (LanguageTool primary + yaspeller fork + Типограф + Vale custom). Style/syntax (отглагольные, пассив, "является", цепочки родительных, кальки) → SKILL.md. No public Russian Vale style pack identified — confirm and possibly bootstrap one.
8. **yaspeller maintenance risk.** Archived 24 April 2024. Phase 2: choose between (a) fork-and-maintain, (b) migrate to LanguageTool 6.6 as primary engine (930 ru rules), (c) keep yaspeller frozen for orthography only and route stylistics via LT + Vale.
9. **Habr LLM-tells reliability.** "Ваш текст воняет GPT" is unbylined and the figures "2–5×" / "1.5–2×" are the Habr author's interpretation of Reinhart et al.'s Biber-feature output, not verbatim PNAS numbers. Cross-check directly against Reinhart et al. PNAS 122(8) e2422455122 (25 Feb 2025) and Kobak et al. *Nature Human Behaviour* 2024–2025 before encoding rules.
10. **Bureau Gorbunov licensing.** "Советы" are CC-unspecified — usable for rule inspiration but not for verbatim copy into SKILL.md.
11. **SKILL.md size constraint.** Anthropic's `skill-creator` documents a budget for SKILL.md instructions. Combining Галь-taxonomy + Контур + Microsoft tone + Ozon dev-doc rules + UI rules + terminology will overflow. Phase 2 must decide: monolithic skill vs split (e.g. `ru-prose-core`, `ru-ui-strings`, `ru-error-messages`).
12. **Code-comment and Javadoc/KDoc/docstring conventions in Russian.** None of the surveyed sources address this directly. Likely needs derivation from JetBrains code-style docs + a small original ruleset.
13. **PR descriptions / review replies.** No public Russian guide identified. Will need original rules informed by Контур "робот vs Контур" persona dichotomy and engineering review etiquette.
14. **License/derivation.** Microsoft PDF is downloadable but proprietary; rules can be paraphrased, not copied. Контур.Гайды and Ozon styleguide are public web pages without explicit OSS licence — same constraint. Phase 2 must produce paraphrased rules with citations.

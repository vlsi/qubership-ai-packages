# Research title

Find proven editorial rules for natural, clear Russian developer documentation and UI text generated by LLMs

# Goal

Run a broad discovery pass to find candidate sources that can help create an LLM skill for writing and editing Russian software-facing prose.

The target is not marketing copy, fiction, SEO text, or generic “humanisation”.
The target is clear, natural, technically precise Russian for software engineering contexts:

- technical documentation
- README files
- API docs
- code comments
- Javadoc/KDoc/docstrings translated or written in Russian
- changelog and release notes
- PR descriptions and review replies
- UI strings and UI translations
- error messages
- log messages intended for humans

The practical goal is to avoid Russian LLM prose that sounds artificial, bureaucratic, over-translated, or machine-generated.

Examples of possible failure modes to investigate:

- excessive long dashes
- кальки from English
- “является”, “осуществляет”, “позволяет”, “данный”, “необходимо”
- overuse of “важно отметить”, “следует учитывать”, “таким образом”
- long noun chains and deverbal nouns
- excessive passive or impersonal constructions
- awkward participial/adverbial-participial phrases
- inconsistent ты/вы/безличный tone
- inconsistent terminology
- literal UI translations
- unnatural button labels or error messages
- mismatch between developer-doc tone and UI tone
- prose that sounds like corporate boilerplate rather than useful documentation

This is Phase 1 only. The goal is discovery and classification, not final skill authoring.

# Important framing

Do not look only for ready-made LLM skills. Ready-made Russian skills may not exist.

Search for reusable rule sources that could later be converted into a compact `SKILL.md`.

It is acceptable to conclude that no single proven Russian LLM skill exists. In that case, identify the strongest reusable rule sources and explain how they could later be combined.

Do not recommend AI-humanizer tools, detector-evasion prompts, paraphrasers, SEO rewriting tools, or generic copywriting prompts.

The goal is better reader experience:

- clarity
- technical precision
- natural Russian
- maintainability
- consistent terminology
- useful UI wording
- reduced native-speaker rejection

# Search scope

Search broadly across these categories.

## 1. Russian software documentation style guides

Look for public style guides from software companies, OSS projects, documentation teams, technical writing teams, and developer platforms.

Prioritize sources that cover:

- developer documentation
- API docs
- README files
- tutorials
- release notes
- error messages
- code comments
- terminology

## 2. Russian UI and localization style guides

Look for Russian localization guidance from:

- Microsoft
- Apple
- Google
- Mozilla
- GitLab
- JetBrains
- Yandex
- VK
- Telegram-related projects if relevant
- open-source localization communities
- translation platforms and localization teams

Focus on:

- UI strings
- button labels
- menu items
- error messages
- placeholders
- tone: ты / вы / neutral
- terminology consistency
- avoiding literal translation from English

## 3. Russian editorial and plain-language sources

Look for reputable sources on clear Russian prose, including:

- Russian editing guides
- technical writing guidance
- plain-language guidance
- style guides from editorial teams
- books or public rule catalogues by named editors
- sources similar in role to “Google/Microsoft style guide”, even if not software-specific

Do not over-prioritize journalism or marketing style unless the rules clearly transfer to technical prose.

## 4. Linters, dictionaries, and mechanical checks

Look for tools and rule sets for Russian prose, spelling, grammar, typographic conventions, and terminology consistency:

- yaspeller
- LanguageTool Russian
- Vale packages for Russian if any
- textlint plugins for Russian if any
- typograf / typography tools
- custom dictionaries
- terminology checkers
- translation QA tools

Classify which rules belong in a linter rather than in an LLM skill.

## 5. Research and serious analysis of Russian LLM prose

Look for academic papers, editor blog posts, technical-writing discussions, and serious comparisons that identify Russian LLM stylistic tells.

Use these only to identify failure modes.
Do not optimize for AI-detector evasion.

## 6. Agent skills, prompts, and rule packs

Look for:

- Claude / agent `SKILL.md` files
- Cursor / Continue rules
- public prompt libraries
- Russian technical-writing prompts
- Russian editing prompts

Treat these as candidates only if they show editorial quality, source reputation, or real-world use.

# What counts as evidence

Treat these as scoring signals, not hard filters:

- authored by a named editor, technical writer, localization team, or documentation team
- adopted by a known software company or large OSS project
- used in a real documentation or localization workflow
- actively maintained
- includes concrete examples
- includes terminology decisions
- has public issue discussions or usage notes
- has a rule set that can be converted into `SKILL.md`
- has a linter/config that can be reused

High star counts, install counts, and downloads are useful supporting evidence, but they must not exclude lower-profile sources with stronger editorial authority.

# Exclusions

Exclude:

- generic marketing-copy prompts
- fiction-writing prompts
- SEO rewriting prompts
- “AI humanizer” tools
- paraphrasers aimed at bypassing AI detectors
- anonymous prompt snippets with no evidence of editorial quality
- grammar-only tools with no style or software-text relevance
- sources that only teach Russian grammar to learners
- sources focused only on academic writing unless directly useful for technical docs


## 1. Executive summary

**Рекомендация: Microsoft APM (Agent Package Manager)** в качестве primary-инструмента — это **единственный OSS-проект**, который одновременно закрывает все восемь must-have требований: управляет семью примитивами (skills, MCP, commands, instructions, agents, hooks, plugins), нативно разворачивает в Claude Code / Cursor / OpenCode / Codex / GitHub Copilot, поддерживает транзитивное разрешение зависимостей с детерминированным `apm.lock.yaml`, тянет пакеты с любого git-хоста (включая self-hosted GitLab, Bitbucket, Azure DevOps), и — что уникально для категории — имеет встроенный supply-chain слой (Unicode/prompt-injection сканер, policy engine, SARIF-выход для GitHub Code Scanning, MCP transitive-trust правила). Лицензия MIT, работает на Windows/macOS/Linux, установка через PyPI/Homebrew/Scoop/native binaries.

**Fallback и гибрид**: APM **не покрывает два слабых места** — (1) отсутствует явная semver range resolution (`^1.2.0`, `~1.2.3`) — только pinning по git ref/tag/SHA, и (2) нет криптографических подписей пакетов. Оба gap закрываются гибридом: **APM + MCPM** (pathintegral-institute/mcpm.sh, MIT) как дополнительный менеджер именно MCP-серверов с зрелым каталогом, **+ Snyk Agent Scan** (ex-mcp-scan, open core) как CI-gate для сканирования skill-файлов на prompt injection и tool poisoning, **+ LiteLLM Claude Code Plugin Marketplace pattern** для приватного self-hosted реестра внутри GitLab. Существующий `REGISTRY.yaml` + Python-резолвер мигрируется в `apm.yml` почти механически — схема APM совместима по семантике с вашим текущим каталогом.

**Ключевой trade-off**: APM — **pre-1.0** (v0.8.11, 6 апреля 2026), community-driven проект, формально перенесённый в `microsoft/` org в феврале 2026 — Microsoft customer support его **не сопровождает** (`SUPPORT.md` явно это декларирует). Bus factor реально ~2 (Daniel Meppiel + Sergio Sisternes из EPAM). Риск breaking changes до релиза 1.0 — реальный. Альтернативы (ARM, Lola, Claude Code native, Vercel skills, SkillKit, OpenSkills) **не проходят по must-have**: либо GPL-3.0/GPL-2.0, либо не поддерживают multi-target, либо нет транзитивных зависимостей + lockfile.

## 3. Детальный разбор top-3

### 3.1 Microsoft APM — primary

**Формальные параметры.** Репозиторий `github.com/microsoft/apm`, docs `microsoft.github.io/apm`, лицензия **MIT**. Создатель и лид-мейнтейнер — **Daniel Meppiel** (@danielmeppiel), соавтор — Sergio Sisternes из EPAM (@sergio-sisternes-epam). Репо перенесён из `danielmeppiel/apm` в `microsoft/apm` **15 февраля 2026** (Discussion #86), v0.7.3 добавил Microsoft OSS-compliance файлы. Текущая версия — **v0.8.11 (6 апреля 2026)**, 40 релизов, ~889 коммитов, язык — Python (95%). Stars — в диапазоне 1.2k–1.7k (активный рост). Установка: `pip install apm-cli`, `brew install microsoft/apm/apm`, Scoop для Windows, native бинарники `aka.ms/apm-unix` / `aka.ms/apm-windows`. Support-канал — только GitHub Issues/Discussions; SUPPORT.md прямо декларирует: «support happens via GitHub Issues/Discussions, not through Microsoft customer support channels».

**Функциональная сверка.** Все восемь must-have закрыты:

1. **Примитивы (7, включая все обязательные)**: instructions (`.instructions.md` с `applyTo` glob), skills (Anthropic `SKILL.md`), prompts (`.prompt.md` — slash-commands), agents (`.agent.md`), hooks (`hooks/*.json`), plugins (`plugin.json`), MCP servers. Источник: microsoft.github.io/apm/reference/primitive-types/.
2. **Targets** — нативный deploy в `.github/`, `.claude/`, `.cursor/`, `.opencode/`, `.agents/skills/` (Codex). Компиляция `AGENTS.md`/`CLAUDE.md` через `apm compile` для Gemini и прочих. **Key design principle**: _«APM does not invent new abstractions — it manages the ones that already exist»_ — сгенерированные файлы остаются нативными, если APM удалить.
3. **Versioned dependencies** — ✅ частично. Поддерживается pinning через `#v1.0.0`, `#v2.0`, `#<branch>`, `#<commit-sha>`; **явной semver range resolution (`^`, `~`) в текущей документации не обнаружено** — refs обрабатываются как точные git references. Транзитивные зависимости отслеживаются с полем `depth: N` в lockfile. `apm.lock.yaml` пинит каждую зависимость к точному commit SHA, содержит `deployed_files` (для безопасной очистки) и `mcp_servers` (для обнаружения устаревших).
4. **Multi-repo sources** — ✅ полностью: _«Any git host: GitHub, GitLab, Bitbucket, Azure DevOps, GitHub Enterprise, self-hosted — if you can `git clone` it, APM can install it. No central registry required»_. Canonical normalization: shorthand, HTTPS, SSH, FQDN — приводятся к единой форме.
5. **Metadata** — формат `apm.yml`; обязательные поля включают `name`, `version`, `dependencies`. JSON Schema доступна в docs.
6. **Marketplace** — CLI: `apm marketplace add/browse/search`; поддерживаются и Copilot CLI, и Claude Code marketplace forматы через `marketplace.json`. Web UI нет в явном виде, но интеграция с Copilot-каталогом.
7. **Security (уникальная сила)** — на install-time сканер скрытых Unicode-символов (tag chars U+E0001–U+E007F, bidi overrides U+202A–U+202E, SMP variation selectors — **blocks install, exit 1**); `apm audit --strip` удаляет, выводы в text/json/**sarif**/markdown. **MCP transitive-trust rule**: MCP-сервера на `depth=1` авто-доверенные, `depth>1` требуют `--trust-transitive-mcp`. **Policy engine** (`apm audit --ci --policy`) — org-level rules, `apm-policy.yaml`, интеграция с GitHub Rulesets.
8. **CI/CD для GitLab self-hosted**: `microsoft/apm-action@v1` официальный GitHub Action; для air-gapped инфра — `PROXY_REGISTRY_URL` / `PROXY_REGISTRY_TOKEN` / `PROXY_REGISTRY_ONLY=1` (Artifactory-style proxy), путь прокси фиксируется в lockfile.

**Реальный пример манифеста:**

```yaml
name: your-project
version: 1.0.0
dependencies:
  apm:
    - anthropics/skills/skills/frontend-design        # skill из subdirectory
    - github/awesome-copilot/plugins/context-engineering
    - github/awesome-copilot/agents/api-architect.agent.md
    - microsoft/apm-sample-package#v1.0.0             # pin to tag
    - git: https://gitlab.acme.corp/platform/coding-standards.git
      path: instructions/security
      ref: v2.0
  mcp:
    - io.github.github/github-mcp-server              # reference to registry
```

**Плюсы.** Единственный инструмент с реальным transitive resolution + lockfile + policy engine в OSS-категории. Real-world security mitigation — документирован incident с Sourcegraph Amp (invisible Unicode prompt injection, embracethered.com), исследование 50 популярных `.cursorrules` показало 6 файлов с hidden characters (issue #312). Air-gapped режим через proxy registry. Активный релиз-цикл, рост звёзд.

**Минусы.** Pre-1.0 — ломающие изменения ожидаемы (в 0.8.x серии добавились marketplace integration, `--global`, Codex support). Bus factor = 2 активных мейнтейнера. Python-зависимость для установки (компенсируется native бинарниками). Нет семверных диапазонов. Нет криптографических подписей пакетов — только Unicode/policy sanity. «Microsoft» в названии — **не про sponsorship и support**, а про OSS-инкубатор.

**Когда выбирать.** Когда требуется единый governance-слой поверх Claude Code + Cursor + OpenCode + Codex, критична воспроизводимость (`apm.lock.yaml`), есть собственная инфра для CI (GitLab) и policy engine нужен. Ваш случай — идеальный fit.

### 3.2 Red Hat Lola — fallback для multi-target через AGENTS.md

**Формальные параметры.** `github.com/RedHatProductSecurity/lola`, лицензия **GPL-2.0-or-later** ⚠️, текущая версия **v0.4.0 (3 февраля 2026)**, последний коммит 26 марта 2026. Stars — **13**, forks — 7, contributors — 6 (Igor Brandao, Katie Mulliken и др., один из них — claude-bot). Python 100%. Установка: `uv tool install git+https://github.com/RedHatProductSecurity/lola`. Companion-проект `RHEcosystemAppEng/agentic-collections` (7 персон, 60+ skills для RHEL/OpenShift/Ansible) — отдельно на Apache-2.0.

**Функциональная сверка.** Закрывает 3/8 must-have полностью: multi-target (Claude Code / Cursor / Gemini CLI / OpenCode через AGENTS.md references), multi-repo sources (любой git URL), YAML-формат marketplace удобен для миграции с REGISTRY.yaml. Не закрывает: транзитивные зависимости и lockfile **отсутствуют полностью**, подписей и sandboxing нет (только path-traversal validation для install hooks), enterprise managed allowlist отсутствует. MCP поддержка упоминается в описании «AI Context Module», но механизм инжекции в `.mcp.json` в README не раскрыт. **Codex не поддерживается.**

**Пример манифеста (`marketplace.yml`):**

```yaml
name: My Marketplace
description: Curated collection of AI skills
version: 1.0.0
modules:
  - name: git-workflow
    description: Git workflow automation skills
    version: 1.0.0
    repository: https://github.com/user/git-workflow.git
    tags: [git, workflow]
    hooks:
      pre-install: scripts/check-python.sh
      post-install: scripts/verify.sh
  - name: monorepo-skills
    repository: https://gitlab.acme.corp/platform/monorepo.git
    path: packages/lola-skills
```

**Лицензионный риск.** GPL-2.0-or-later на CLI — если Lola используется как **stand-alone инструмент** (не встроенный, не линкуемый в проприетарный код), copyleft не «заражает» продукт. Но **обязателен legal review** перед embedding, fork становится copyleft-бременем.

**Когда выбирать.** Если APM не подходит по некоторой причине (например, требуется pure Red Hat stack с OpenShift integration), а multi-target критичнее supply-chain дисциплины. Для вашего кейса — **не primary**, потенциально компонент гибрида для AGENTS.md-first экосистем.

### 3.3 Claude Code native plugin marketplaces — дополнительный слой для Claude-heavy команд

**Формальные параметры.** Часть Anthropic Claude Code (proprietary CLI, но плагин-формат открыт и совместим с marketplace manifest schema `https://anthropic.com/claude-code/marketplace.schema.json`). Plugin-механизм созрел в 2026: появился `strictKnownMarketplaces` allowlist для managed enterprise settings, cache isolation (`~/.claude/plugins/cache`), path-traversal защита, `--keep-data` при uninstall. В CHANGELOG конца 2025 — начала 2026: _«Fixed plugin install not honoring dependencies declared in `plugin.json` when the marketplace entry omits them; `/plugin install` now lists auto-installed dependencies»_ — **транзитивный resolver появился, но не задокументирован публично**, issue #9444 по shared «common-core» библиотекам ещё open.

**Функциональная сверка.** Примитивы — полная поддержка (skills, MCP, commands, subagents, hooks, LSP, monitors, output styles). Targets — **только Claude Code**. Multi-repo — любой git URL (`/plugin marketplace add https://gitlab.corp/plugins.git#v1.0.0`). Managed enterprise controls — сильнейшие из всей выборки: `strictKnownMarketplaces` allowlist, `extraKnownMarketplaces` для team-provisioning, managed scope фиксирует набор плагинов, auto-update OFF для third-party по умолчанию. **Lockfile отсутствует. Подписей нет.** Anthropic явно декларирует: _«Anthropic does not control what MCP servers, files, or other software are included in plugins and cannot verify that they work as intended»_.

**Пример `.claude-plugin/marketplace.json`:**

```json
{
  "$schema": "https://anthropic.com/claude-code/marketplace.schema.json",
  "name": "acme-internal-marketplace",
  "version": "1.0.0",
  "owner": { "name": "Acme Platform", "email": "platform@acme.corp" },
  "plugins": [
    {
      "name": "acme-devops-tools",
      "source": { "source": "url", "url": "https://gitlab.acme.corp/ai/devops-plugin.git" },
      "version": "2.1.0",
      "strict": true
    }
  ]
}
```

**Когда выбирать.** Для корпоративных команд, использующих **только Claude Code**, с жёстким governance через managed settings — родной стек Anthropic. В вашем multi-target кейсе — **не primary**, но APM нативно генерирует этот формат и интегрируется с ним, так что используется как deployment target.

## 5. Гибридные архитектуры

Поскольку ни один OSS-инструмент не закрывает **всё** идеально, рекомендуется гибрид из 3–4 компонентов с APM как orchestration-слоем.

**Рекомендуемая архитектура для вашего контекста:**

|Слой|Инструмент|Роль|
|---|---|---|
|Orchestration + transitive resolution + lockfile|**Microsoft APM**|Source of truth, `apm.yml` + `apm.lock.yaml`, deploy во все таргеты|
|Внутренний реестр пакетов|**GitLab self-hosted** + **LiteLLM marketplace pattern** (optional Admin UI)|Приватный git-backed source; `PROXY_REGISTRY_URL` для air-gapped|
|MCP-specific каталог|**MCPM** (mcpm.sh) **+ MCP Registry** (community subregistry)|Дополнительный источник MCP-серверов с зрелым community каталогом; приватная subregistry как mirror|
|Supply chain scanning|**Snyk Agent Scan** (ex Invariant mcp-scan) + **APM Unicode sanitizer**|Pre-install CI gate: prompt injection, malware payloads, hidden unicode, tool poisoning|
|Runtime observability|Claude Code `OpenTelemetry` + Prometheus + VictoriaMetrics|Audit trail для agent invocations (gap по стандарту MCP 2026)|
|Policy enforcement|`apm audit --ci --policy apm-policy.yaml` + GitLab CI required job|Org-level allowlist/denylist, MCP transport controls, manifest requirements|

**Потоки данных.** `apm.yml` в репозитории → GitLab CI запускает `apm audit --ci --policy` + `snyk-agent-scan` → при успехе `apm install` на машине разработчика или в контейнере → deploy в `.claude/`, `.cursor/`, `.opencode/`. MCP-сервера либо через `apm.yml` (для zainfra-managed), либо через MCPM для ad-hoc discovery. `apm.lock.yaml` commitится в git для reproducibility.

**Альтернативный минимальный гибрид** (если не хочется вводить MCPM/Snyk сразу): APM + встроенный `apm audit` + проксирование через Artifactory → получается самодостаточный стек, supply-chain на уровне hidden-unicode + policy engine.

## 6. Рекомендация и план адаптации

**Выбор: Microsoft APM как primary, последующая постепенная интеграция Snyk Agent Scan в CI и MCPM как complementary каталог.**

**План миграции с `REGISTRY.yaml` + Python-резолвера:**

1. **Неделя 1–2 — pilot.** На одном тестовом репозитории: установить `apm-cli`, перевести 3–5 skills из текущего REGISTRY.yaml в `apm.yml`-формат. Проверить `apm install` в Claude Code / Cursor / OpenCode локально. Сравнить поведение с текущим Python-резолвером на предмет транзитивных зависимостей.
2. **Неделя 3–4 — registry setup.** Поднять internal GitLab-группу как primary source (`apm` dependencies → `git: https://gitlab.acme.corp/...` entries). Настроить `PROXY_REGISTRY_URL` через Artifactory для air-gapped узлов. Сгенерировать `apm-policy.yaml` с org-level allowlist: разрешённые источники только `github.com/anthropics/`, `github.com/microsoft/`, `gitlab.acme.corp/*`.
3. **Неделя 5–6 — CI integration.** `microsoft/apm-action@v1` (если используется GitHub) или эквивалент на GitLab CI: `apm audit --ci --policy --output sarif` как required status check. Добавить Snyk Agent Scan как второй gate (`uvx snyk-agent-scan@latest`).
4. **Неделя 7–8 — migration at scale.** Автогенерация `apm.yml` из текущего REGISTRY.yaml скриптом (формат YAML → YAML преобразуется прямолинейно). Все 300+ компонентов переведены, каждый с `apm.lock.yaml` в git.
5. **Месяц 3+** — раскатка на все команды через managed `.claude/settings.json` с `strictKnownMarketplaces`.

**Gaps APM и компенсация:**

|Что не покрывает APM|Как компенсировать|
|---|---|
|Semver ranges (`^`, `~`) — только точные refs|Использовать **tag-based pinning** (`#v1.2.0`) + регулярный `apm update` workflow с review. При необходимости написать wrapper-скрипт, транслирующий range → latest-matching-tag перед `apm install`|
|Криптографические подписи пакетов|Layer Snyk Agent Scan + Sigstore/Cosign подписывание собственных internal skills на стороне GitLab CI; валидация через custom hook в `apm audit`|
|MCP server registry discovery|MCPM или прямой reference через MCP Registry (`io.github.owner/name` нотация уже поддерживается APM в `mcp:` секции)|
|Web UI для marketplace|LiteLLM marketplace Admin UI как self-hosted фронт; Claude Code native `/plugin` интерактивный UI для end-users|
|Runtime prompt injection|Invariant Guardrails (коммерческий) или custom hooks с Parry (community) на tool I/O|
|Отсутствие Microsoft support|Активно участвовать в GitHub Discussions, рассмотреть финансирование через GitHub Sponsors для bus-factor mitigation|

**Риски на горизонте 12–24 месяца.**

1. **APM pre-1.0 → 1.0 breaking changes** — реалистичная угроза. Митигация: следить за CHANGELOG, закрепить v0.8.x для production через proxy registry, регулярно тестировать upgrade path на staging.
2. **Конкуренция от `gh skill`** — GitHub 16 апреля 2026 анонсировал official extension, использует agentskills.io spec и provenance pinning. На горизонте 12 месяцев может стать **де-facto standard** для skill distribution благодаря GitHub distribution и CI integration. Митигация: мониторить; APM уже compatible с agentskills.io spec, а значит skills переносимы обратно при необходимости pivot.
3. **Anthropic native plugin marketplace стабилизируется** — документирование transitive resolver, появление lockfile, расширение target-поддержки (unlikely, но наблюдать). Митигация: APM нативно deploys в `.claude-plugin/` формат, так что двуязычие будет органичным.
4. **Стандарты AGENTS.md + SKILL.md под Linux Foundation (AAIF, с 9 декабря 2025)** — консолидация продолжится, fragmentation снизится. Это **хорошо для APM** как vendor-neutral менеджера.
5. **Supply chain incidents будут расти** — на апрель 2026 уже зарегистрированы CVE-2025-6514 (mcp-remote RCE, CVSS 9.6), CVE-2025-59528 (Flowise RCE, CVSS 10.0), CVE-2025-59536 / CVE-2026-21852 (Claude Code config-driven RCE, fixed в 2.0.65+), «ClawJacked» CVE-2026-28363 (OpenClaw DNS-rebinding), ToxicSkills campaign (Snyk сообщает о **36.82% uploaded skills с security flaws, 1467 malicious payloads**), «Claudy Day» data exfiltration в default Claude.ai sessions. Supply-chain слой (Snyk Agent Scan + Unicode sanitizer + policy engine) **не опционален**.
6. **Bus factor риск APM** — 2 активных мейнтейнера для проекта с Microsoft-брендингом. Возможные сценарии: (а) расширение команды через Microsoft, (б) community-fork при уходе мейнтейнеров, (в) слияние с `gh skill` при консолидации стандарта. Митигация: внутренняя компетенция на чтение APM-кода (Python, ~5K LOC), готовый fallback на Lola как ставный vendor-neutral multi-target-менеджер.

## Заключение

**Основной вывод, отличающий этот анализ от поверхностного сравнения по звёздам:** на апрель 2026 **Microsoft APM — единственный OSS-кандидат с транзитивными зависимостями и детерминированным lockfile**, которые требование №3 делает блокером. Остальные инструменты с большей популярностью (Vercel skills 7.5k ⭐, OpenSkills 8.8k ⭐) на этом требовании проваливаются — они либо симлинкуют, либо pull-latest из main-ветки, что несовместимо с supply-chain дисциплиной для 300+ компонентов.

**Второй контр-интуитивный вывод**: GPL-лицензия Lola — не теоретический риск, а реальный блокер для корпоративного адаптера, планирующего embedding или custom-расширения; «Red Hat Product Security» в названии не гарантирует enterprise-grade security — подписей и sandbox нет у обоих кандидатов.

**Горизонт наблюдения.** `gh skill` от GitHub (16 апреля 2026) и официальная Anthropic plugin dependency resolver — два вектора, которые в течение 12 месяцев могут сдвинуть баланс. Выбор APM с гибридом MCPM + Snyk Agent Scan — оптимальная точка входа: APM нативен с agentskills.io spec, компилирует AGENTS.md/CLAUDE.md в стандартные форматы, генерирует `.claude-plugin/marketplace.json` — что обеспечивает **lossless миграцию** на любой будущий стандарт без потери контента. Дата последней проверки данных — **17 апреля 2026**.

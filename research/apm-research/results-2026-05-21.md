# Open-Source Skill / Plugin / Package Managers for AI Coding Agents — Corporate Evaluation

**Verification date: 21 May 2026.** All numeric data (stars, releases, license) verified against current GitHub state on this date unless otherwise noted.

---

## TL;DR

- **Adopt Microsoft APM (`github.com/microsoft/apm`, MIT, v0.13.0, 2.4k stars / 180 forks) as the primary skill/plugin/package manager.** It is the only candidate that simultaneously manages all five primitive types (skills, MCP servers, commands, AGENTS.md/CLAUDE.md instructions, agents/sub-agents), targets Claude Code + Cursor + OpenCode + Codex + Copilot + Gemini + Windsurf from one manifest, supports transitive dependencies + lockfile, multi-host git (GitHub/GitLab/Bitbucket/Azure DevOps/GitHub Enterprise/Gitea/Gogs), and ships an enterprise policy file with allow-listing and a hidden-Unicode supply-chain scanner.
- **Treat `vercel-labs/skills` (`npx skills`, Apache-2.0, 19.4k stars / 1.6k forks) as a fallback/discovery layer**, and let APM consume the same `skills.sh` sources it indexes. Optionally complement APM with `mcpm.sh` or `ravitemer/mcp-hub` for advanced MCP runtime sharing, and `intellectronica/ruler` (MIT, 2.6k stars / 134 forks) only if APM's AGENTS.md compile step proves insufficient for an edge case.
- **One important caveat against APM (req. #3, "ideal: semver ranges"):** APM does **not** implement `^1.2.0` / `~1.2.3` range resolution. Per the maintainer on `renovatebot/renovate#42507`: *"APM does not support version ranges. Dependencies pin to an exact ref (tag, branch, or commit SHA). When no ref is specified, APM resolves to the latest default-branch commit and pins the exact SHA in the lock file."* This satisfies the brief's "minimum acceptable" tier (pinned refs + lockfile + transitive resolution) and is the same model used by Helm/Argo for image tags — but if `^`-style ranges are non-negotiable, no surveyed tool implements them.

## Key Findings

1. **The "AI agent package manager" category exploded in the last 6 months.** Microsoft APM (initial release late 2025) reached v0.13.0 in May 2026 with ~50 releases; Vercel announced `skills` and `skills.sh` on 20 January 2026 ("Introducing skills, the open agent skills ecosystem" — Vercel changelog); Anthropic officially launched its Claude Code plugin marketplace on 16 December 2025 ("We're introducing a first-party plugins marketplace" — @claudeai); GitHub shipped `gh skill` in GitHub CLI v2.90.0 on 16 April 2026 (github.blog changelog). The space is *new*, not stable.
2. **Only one tool surveyed actually behaves like `npm`/`pip`** — i.e., has a declarative manifest, lockfile, transitive resolution, multi-host registries, and a policy/audit layer. That is Microsoft APM. Every other tool surveyed is either a *file copier with discovery* (`npx skills`, `gh skill`, OpenSkills, SkillKit, Lola), a *single-source-of-truth distributor* (Ruler), a *catalogue-only website* (`skills.sh`, `claudemarketplaces.com`, `aitmpl.com`), an *agent-specific plugin runtime* (Claude Code marketplace, OpenCode native skills, Continue.dev hub), or a *MCP-runtime manager only* (mcpm.sh, mcp-hub, mcphub Python, mcp-get — the last is archived).
3. **License risk is concrete and important.** Two of the named "known candidates" in the brief are copyleft. `RedHatProductSecurity/lola` and its active fork `LobsterTrap/lola` both ship under **GPL-2.0-or-later** (verbatim from each README's License section). `colchuck-ai/ai-resource-manager` (formerly `jomadu/ai-resource-manager`, "ARM") ships under **GPL-3.0** with the full text in `LICENSE.txt` (verified by direct fetch). Per the brief's licence rule both are excluded; APM (MIT), `vercel-labs/skills` (Apache-2.0), `intellectronica/ruler` (MIT), `numman-ali/openskills` (Apache-2.0), `rohitg00/skillkit` (Apache-2.0), `pathintegral-institute/mcpm.sh` (MIT) and `ravitemer/mcp-hub` (MIT) are business-friendly.
4. **No surveyed tool implements semver-range resolution between skill packages.** APM, Lola, gh skill, OpenSkills, SkillKit and ARM all pin to exact refs (git tags or commits). ARM's marketing copy says "semantic versioning" but its README does not demonstrate `^`/`~` syntax. Lola's `.lola-req` shows `>=` and `@ref` syntax — closer to a `requirements.txt` than a true resolver. This is a category-wide gap, not an APM-specific one.
5. **Multi-target coverage of the four required agents (Claude Code, Cursor, OpenCode, Codex) is rare.** APM ✅, `vercel-labs/skills` ✅ (40+ agents), `gh skill` ✅ (35+ hosts per GitHub blog), `intellectronica/ruler` ✅ (exactly 32 targets — agentsmd, aider, amazonqcli, amp, antigravity, augmentcode, claude, cline, codex, copilot, crush, cursor, factory, firebase, firebender, gemini-cli, goose, jetbrains-ai, jules, junie, kilocode, kiro, mistral, opencode, openhands, pi, qwen, roo, trae, warp, windsurf, zed, per the `--revert` flag in the README). Lola supports Claude Code + Cursor + Gemini CLI + OpenCode but **not Codex**. ARM supports Cursor + Copilot + Amazon Q + Kiro but **none of the four required targets**. The Claude Code native marketplace is by definition Claude-Code-only.
6. **Pre-1.0 maturity is the norm.** APM v0.13.0; Lola v0.4.3; ARM v5.0.0 (but maintainer publicly admits v1→v3 was "nuke and pave… honestly we've probably made [poor design choices] in v3 too" — direct README quote, a red flag). OpenSkills 1.5.0; SkillKit 1.24.0; `vercel-labs/skills` 1.5.7. All require version-pinning of the manager itself.
7. **Supply-chain controls matter for your context.** The brief explicitly cites prompt-injection via skill files as a real risk. APM's `apm install` and `apm audit` run a hidden-Unicode / tag-character / bidi-override scanner by default ("Every install scans for hidden Unicode and other tag-character / bidi-override patterns before deployment. Critical findings block the package; the install exits 1."), gates transitive MCP servers behind `--trust-transitive-mcp`, records per-file content hashes in `apm.lock.yaml`, and lets a security team write `apm-policy.yml` that constrains sources/scopes/primitives across enterprise → org → repo. No other surveyed tool offers this depth of supply-chain story. `gh skill` adds content-addressed change detection that an APM hybrid can consume.

### Top-3 detailed breakdown

#### 1) Microsoft APM — **Primary recommendation**

**Identity.** `github.com/microsoft/apm`. Created and lead-maintained by Daniel Meppiel (@danielmeppiel, Microsoft) and Sergio Sisternes (@sergio-sisternes-epam, EPAM). Python (~97%). **License: MIT** (per GitHub-rendered sidebar; link target `/microsoft/apm/blob/main/LICENSE`).

**Versioning at verification date.** Latest release **v0.13.0** (per the closed `0.13.0` milestone dated 11 May 2026 on `github.com/microsoft/apm/issues`), following v0.12.4 (07 May 2026), v0.12.0, v0.11.0 (27 Apr 2026) and v0.10.0. ~50 releases total. **2.4k stars / 180 forks** per the issues page header (May 2026, growing from 171 forks on the May 11 milestone page). The project is pre-1.0 — flag as maturity risk; breaking defaults have shipped (e.g., v0.12.0 flipped `apm pack`'s default `bundle-format` from `apm` to `plugin`, requiring downstream callers to set `bundle-format: 'apm'` to restore the old layout).

**Manifest example (verbatim from `microsoft/apm` README).**

```yaml
# apm.yml — ships with your project
name: your-project
version: 1.0.0
dependencies:
  apm:
    - anthropics/skills/skills/frontend-design                # skill
    - github/awesome-copilot/plugins/context-engineering      # plugin
    - github/awesome-copilot/agents/api-architect.agent.md    # single agent
    - microsoft/apm-sample-package#v1.0.0                     # full package, pinned to tag
  mcp:
    - name: io.github.github/github-mcp-server
      transport: http
```

`apm.lock.yaml` records resolved source URL, resolved git commit SHA and per-file content hashes — the supply-chain provenance the brief requires.

**Functional compliance.**

**Architecture for an infra/ops shop.**
- **Conflict handling.** Local files always win against installed package files (docs: *"Local files automatically override dependency files with same names"*). Stale-file cleanup is gated by per-file content hashes recorded in the lockfile.
- **Lockfile / reproducibility.** `apm.lock.yaml` pins resolved sources + content hashes; `--frozen` makes `apm install` refuse if the lockfile is incomplete; `apm audit` rebuilds context in scratch and diffs against the working tree to catch hand-edits, drift-detection enabled by default since v0.11.x.
- **Private registry + auth.** APM delegates non-GitHub auth to `git` (SSH keys, deploy tokens, PATs work natively); for GitHub it tries `gh auth token` then falls back to `git credential fill`; Azure DevOps gets a PAT→bearer fallback (`#1214`). MCP registry overridable per-server via `--registry`, env var `MCP_REGISTRY_URL`; unreachable custom registries fail closed.
- **Supply-chain security.** Hidden-Unicode / tag-character / bidi-override scanner runs on every install + on demand via `apm audit`; transitive MCP servers gated behind `--trust-transitive-mcp`; `apm-policy.yml` (with tighten-only inheritance enterprise → org → repo) allow-lists sources, scopes and primitives at install time; air-gapped CI playbook documented. PyPI artefacts are signed with attestations under `microsoft/apm` build-release.yml publisher.
- **Migration from `REGISTRY.yaml + Python resolver`.** Mechanically simple: APM is Python, `apm.yml` is YAML, your existing skills can stay in a local `.apm/` folder verbatim; `apm pack --format plugin` exports a Claude Code plugin so the same package ships simultaneously to APM consumers and to the Anthropic marketplace.

**Adoption.** Companion GitHub Action `microsoft/apm-action` (v1.7.x), sample package `microsoft/apm-sample-package`, Scoop bucket `microsoft/scoop-apm`, doc site `microsoft.github.io/apm/`. External contributors in v0.10.x–v0.13.x release notes include @stbenjam (Red Hat), @JanDeDobbeleer (Oh My Posh), @awakecoding, @qorexdevs, @chaobo8484, @guwenqing, @mvanhorn, @ganesanviji (added Gitea), @edenfunf, @coakenfold. Bus factor is two named maintainers + an active external base.

**Pros.** True package-manager semantics — manifest, lockfile, transitive deps, multi-host, policy, signed builds. The only tool surveyed that ships every box.
**Cons.** Pre-1.0; no semver range syntax; marketplace federation still WIP; learning curve around `.apm/` vs `apm_modules/` vs `.github/` vs `.claude/` directories; trademark-bound under Microsoft brand guidelines (cannot ship modified forks under the same name).

1. **APM is the dependency manager.** All skills, MCP servers, agents, commands and instructions enter the org via `apm.yml`. `apm.lock.yaml` is committed.
2. **`apm-policy.yml` is enforced at the org level**, gated by `apm audit --ci` in GitLab CI; Artifactory acts as proxy mirror for the air-gapped enclave.
3. **`npx skills` and `gh skill`** remain available for individual developers' discovery; any kept skill is promoted to a line in the shared `apm.yml`.
4. **MCP runtime sharing**: when ≥10 developers share long-running MCP servers, delegate runtime fan-out to `mcpm.sh` or `mcp-hub`. APM writes the configuration; the hub serves it.
5. **Renovate** auto-bumps git tag refs (`#vX.Y.Z`); APM regenerates the lockfile. Until `renovatebot/renovate#42507` ships the dedicated apm.yml datasource, wrap `apm outdated --json` for a custom Renovate config.

### Recommendation and adaptation plan

**Choose Microsoft APM as the primary skill/plugin/package manager.**

**12-week rollout (~300 components, infra/ops scope).**

| Week | Milestone | Success criterion |
|---|---|---|
| 1–2 | Pilot one team (≤10 components). Internal GitLab mirror of `microsoft/apm-sample-package` + one of your own skills repos. Install `apm` via Scoop / pip / curl. | `apm install` works against self-hosted GitLab; one developer gets configured Claude Code + Cursor + OpenCode + Codex in one command. |
| 3–4 | Convert existing `REGISTRY.yaml` + Python resolver to `apm.yml` + `.apm/`. Each skill becomes either a standalone APM package (own repo) or a `.apm/skills/<name>/` entry. | Bit-for-bit equivalent agent configuration emitted to all four targets. |

**What APM does not satisfy, and how to compensate.**

- **Semver ranges (req. #3 ideal tier).** Compensation: pin to git tags (`#vX.Y.Z`); let Renovate or cron-driven `apm outdated --json` propose bumps. This is the Helm/Argo image-tag model. If `^1.2.0` ranges are non-negotiable, raise an upstream issue — no active workstream as of v0.13.0.
- **Web marketplace UI.** APM has a doc site, not a browse-search-install web app. Use `skills.sh` for discovery; mirror an index of allow-listed packages onto an internal static site (data is just `apm.yml` files in known repos).
- **Pre-1.0 maturity.** Pin the APM CLI version in `microsoft/apm-action` (`apm-version: '0.13.0'`) and the Scoop bucket; do not auto-upgrade. Maintain an internal mirror of release tarballs.
- **Bus factor.** Two named maintainers + growing external base, but young. Vendor a frozen copy into your GitLab; keep working knowledge of the codebase (~30K Python LOC); MIT license means a fork is always an option.

**12–24-month risks.**

- **Standard drift (high probability, manageable):** Agent Skills, AGENTS.md, MCP specs all evolve. APM tracks each; follow `microsoft.github.io/apm/changelog`.
- **Vendor consolidation (medium probability, medium impact):** `gh skill` (GitHub CLI v2.90.0, 16 Apr 2026 release) is positioning to own the skill subset. APM is designed to consume `gh skill publish` GitHub releases — alliance not competition. Monitor strategy.
- **Project abandonment (low probability):** APM is increasingly relied on inside Microsoft (Microsoft 365 Copilot Cowork features in v0.10.x, GitHub Action under official MS org). Keep the vendored copy.
- **Licence/governance shift:** MIT under `microsoft/`. Trademark guidelines block confusing modified releases but do not restrict use; no realistic licence-shift risk.
- **Specific feature regressions:** Pre-1.0 — expect breakage. Pin and test.

## Recommendations

1. **Adopt `microsoft/apm` (v0.13.0, MIT) as the primary skill/plugin/package manager.** Commit `apm.yml` + `apm.lock.yaml`; enforce `apm-policy.yml` org-wide; gate CI on `apm audit --ci`. Roll out in three waves over ~12 weeks per the plan above. *Benchmark that would change this:* APM ceases active development for >90 days, or its license changes from MIT, or a v1.0 successor with `^/~` semver-range support emerges from another well-resourced org (Vercel `skills` v2+, GitHub `gh skill` evolving into a manifest-based tool, or a hypothetical Anthropic-sponsored cross-agent package manager).
2. **Pin the APM CLI itself.** Use `apm-action@v1.7.x` with `apm-version: '0.13.0'`. Re-evaluate every minor release.
3. **Keep `vercel-labs/skills` and `gh skill` as discovery / one-shot tools, never as the source of truth.** Any skill a developer wants to keep moves into `apm.yml`. *Benchmark:* if `skills.sh` adds a manifest layer with transitive deps, re-evaluate as a possible co-primary.
4. **Treat the Claude Code native plugin marketplace as a publishing target, not a source of truth.** Use `apm pack --format plugin` to ship internal skills there.
5. **Do not adopt Lola or ARM.** Both are copyleft (GPL-2.0-or-later and GPL-3.0 respectively) — the brief's licence rule excludes them. Even if your legal team accepts copyleft for tooling, both fail other hard requirements (Lola: no Codex target, no transitive deps; ARM: none of the four required agents, no skills/MCP/agents/instructions).
6. **Build a 6-month re-evaluation cadence into the adoption plan.** This category did not exist a year ago and may consolidate or fragment significantly. Calendar a review for November 2026 and May 2027.
7. **Contribute upstream.** Two specific gaps where your team would have leverage:
   - `renovatebot/renovate#42507` — apm.yml datasource. Already a maintainer-acknowledged design.
   - Semver-range support in APM itself — currently zero-cost to land if someone wants to build it; no surveyed competitor has it either, so the first to ship gains a moat.

## Caveats

1. **GitHub star/fork counts for `microsoft/apm` differ between rendered pages** (2.4k stars / 180 forks on the issues page, May 2026, growing from 171 forks on the May 11 milestone page; ~930/77 on older cached pages). The higher figure is current.
2. **Codex compatibility is partial.** APM acknowledges: *"Codex CLI currently does not support remote MCP servers; the install will skip Codex with a notice."* Lola does not support Codex at all (supported-assistants table lists Claude Code, Cursor, Gemini CLI, OpenCode only).
3. **Two of the brief's named candidates are GPL.** Lola = GPL-2.0-or-later (verbatim from each repo's README License section); ARM = GPL-3.0 (full text in `LICENSE.txt`). The brief flags GPL/AGPL as a risk; both are excluded from the recommendation. The *outputs* (rendered prompts, AGENTS.md content) are not GPL-encumbered, but the *manager itself* is — meaning organisational modifications would need to be redistributed if you ship a modified build.
4. **Pre-1.0 is real for every primary candidate.** APM has shipped breaking defaults between v0.10 and v0.12 (bundle format default flip; skills moved to `.agents/skills/`). Pin the CLI.
5. **Marketplace.json schemas are inconsistent.** The Anthropic-published `https://anthropic.com/claude-code/marketplace.schema.json` URL returns 404 at verification date (multiple open GitHub issues). The actual schema is documented in prose; the third-party `https://json.schemastore.org/claude-code-plugin.json` exists and is what APM emits to.
6. **No tool surveyed implements `^1.2.0` semver-range resolution between skill packages.** This is a category-wide gap, not an APM-specific limitation. ARM claims "semantic versioning" but its README does not demonstrate `^/~` syntax; APM, Lola, gh skill, OpenSkills, SkillKit and the Claude marketplace all use exact-pin models.
7. **The ecosystem is moving extremely fast.** Most of these tools did not exist 6 months ago: Anthropic's official Claude Code plugin marketplace launched 16 December 2025; Vercel's `skills.sh` launched 20 January 2026; `gh skill` shipped in GitHub CLI v2.90.0 on 16 April 2026; APM hit v0.13.0 on 11 May 2026. Plan for re-evaluation every 6 months at minimum.
8. **Source-of-truth lookups for fork counts and star counts vary by which GitHub view is rendered** (issues page vs releases page vs profile page can return slightly different numbers due to caching). The figures cited here are the most recent available at verification date.

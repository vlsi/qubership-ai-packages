## Цель
Выбрать один или несколько (в случае гибридной архитектуры) open-source 
инструмента для управления skills, MCP серверами, агентами/субагентами, 
командами и инструкциями (contributions к AGENTS.md / CLAUDE.md) в corporate-окружении.

## Жёсткие требования (must-have)

1. **Примитивы.** Менеджер должен управлять как минимум:
   - Skills (обязательно)
   - MCP servers (желательно, но не блокер)
   - Commands / slash-commands
   - Instructions — contributions в AGENTS.md / CLAUDE.md
   - Agents / Subagents

2. **Targets.** Установка пакетов в известные агенты минимум:
   - Claude Code
   - Cursor
   - OpenCode
   - Codex (OpenAI)

3. **Versioned dependencies.** Пакеты должны объявлять явные 
   версионированные зависимости на другие пакеты. Идеально — semver с 
   range resolution (^1.2.0, ~1.2.3), минимум — pinned refs с lockfile 
   и transitive resolution.

4. **Multi-repo sources.** Возможность тянуть пакеты из нескольких 
   репозиториев одновременно: публичных (GitHub OSS) и корпоративных 
   (self-hosted GitLab, Bitbucket, Azure DevOps, приватный Gitea).

5. **Metadata.** Каждый пакет (skill) должен содержать обязательно: 
   `name`, `description`, `version`, `author`, `dependencies`. 
   Наличие официальной схемы валидации — плюс.

6. **Marketplace.** Способ листинга и поиска пакетов:
   - CLI — обязательно
   - Web UI — nice to have

7. **Лицензия.** MIT, Apache 2.0 или другая business-friendly 
   (BSD, MPL-2.0, ISC). GPL/AGPL — исключить или явно обозначить как риск.
   
8. **OS Support**: Windows, macOS, Linux

## Критерии оценки каждого кандидата

Для каждого найденного инструмента собери:

**Формальные параметры:**
- Название, URL, мейнтейнер (организация/автор)
- Лицензия (точное название)
- Текущая версия и дата последнего релиза
- GitHub stars / forks / open issues / contributors
- Частота коммитов за последние 3 месяца
- Язык реализации

**Функциональная сверка (по 7 требованиям выше):**
- Для каждого требования: ✅ полностью / ⚠️ частично / ❌ нет
- Конкретная цитата из документации или код-ссылка, подтверждающая оценку
- Если частично — что именно не хватает

**Архитектура:**
- Формат манифеста (apm.yml / package.json / custom)
- Как устроен dependency resolution
- Как обрабатываются конфликты (одинаковые имена из разных источников)
- Lockfile формат и воспроизводимость
- Поддержка приватных регистров и аутентификации
- Security: сканирование пакетов, supply chain защита, подписи

**Adoption & ecosystem:**
- Сколько публичных пакетов/skills доступно
- Крупные adopters (если есть публичные упоминания)
- Совместимость с открытыми стандартами (AGENTS.md, Agent Skills spec, MCP)
- Активность сообщества, наличие governance-модели

**Риски:**
- Vendor lock-in (даже для OSS инструментов)
- Bus factor (сколько активных мейнтейнеров)
- Зрелость (< 1.0 versions — риск breaking changes)
- Известные security-инциденты или уязвимости

## Формат output

### 1. Executive summary
2-3 абзаца: какой инструмент рекомендован как primary, какой как fallback, 
ключевые trade-offs.

### 2. Сравнительная таблица
Все кандидаты × 7 требований + лицензия + популярность. 
Используй ✅/⚠️/❌ + числовые метрики где применимо.

### 3. Детальный разбор top-3 кандидатов
Для каждого — все формальные и функциональные параметры, цитаты из 
документации, реальный пример apm.yml / манифеста, плюсы, минусы, 
когда выбирать.

### 4. Short-list дисквалифицированных
Кто был рассмотрен и отброшен — с конкретной причиной (нарушает 
требование №X, GPL-лицензия, заброшен и т.д.).

### 5. Гибридные архитектуры
Если один инструмент не закрывает все требования — предложи 
комбинации (например, "APM для orchestration + npx skills для установки 
из широкого каталога").

### 6. Рекомендация + план адаптации
- Какой инструмент выбрать
- Какие требования он не покрывает и как это компенсировать 
  (кастомные расширения, обёртки, миграция)
- Риски на горизонте 12-24 месяцев (убывающий proj, смена стандартов)

## Anti-bias инструкции

- Не выбирай инструмент по популярности — 7.5k stars не перевешивают 
  нарушение требования №3.
- Не принимай README на веру — проверяй заявленные фичи по коду или 
  документации.
- Если нашёл расхождение между README и реальной реализацией — отметь 
  это явно.
- Если требование выполнено через "планируется в roadmap" — это ❌, 
  не ✅.
- Для каждой рекомендации укажи дату последней проверки — индустрия 
  быстро меняется.

## Контекст adopter'а (для калибровки рекомендаций)

- Corporate окружение, infrastructure/ops команда
- Используется Kubernetes, Helm, ArgoCD, GitLab (self-hosted возможен)
- Масштаб: 300+ компонентов, параллельное использование Claude Code, 
  OpenCode, Cursor
- Уже существует каталог internal skills с явными зависимостями 
  (REGISTRY.yaml + Python-резолвер) — миграция с него должна быть 
  реалистичной
- Критичен supply chain: пакеты будут читаться агентами, 
  prompt-injection через skill-файлы — реальный риск

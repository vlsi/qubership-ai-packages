
## 1. Executive summary

**Recommendation: Microsoft APM (Agent Package Manager)** as the primary tool — it is the **only OSS project** that simultaneously meets all eight must-have requirements. It manages seven primitives (skills, MCP, commands, instructions, agents, hooks, plugins), deploys natively into Claude Code / Cursor / OpenCode / Codex / GitHub Copilot, supports transitive dependency resolution with a deterministic `apm.lock.yaml`, pulls packages from any git host (including self-hosted GitLab, Bitbucket, Azure DevOps), and — uniquely for the category — ships a built-in supply-chain layer (Unicode / prompt-injection scanner, policy engine, SARIF output for GitHub Code Scanning, MCP transitive-trust rules). Licence is MIT, it runs on Windows / macOS / Linux, and installs via PyPI / Homebrew / Scoop / native binaries.

**Fallback and hybrid**: APM **does not cover two weak spots** — (1) there is no explicit semver range resolution (`^1.2.0`, `~1.2.3`); only pinning by git ref / tag / SHA, and (2) there are no cryptographic package signatures. Both gaps are closed by a hybrid: **APM + MCPM** (pathintegral-institute/mcpm.sh, MIT) as an additional manager dedicated to MCP servers with a mature catalogue, **+ Snyk Agent Scan** (ex-mcp-scan, open core) as a CI gate that scans skill files for prompt injection and tool poisoning, **+ LiteLLM Claude Code Plugin Marketplace pattern** for a private self-hosted registry inside GitLab. The existing `REGISTRY.yaml` + Python resolver migrates to `apm.yml` almost mechanically — the APM schema is semantically compatible with your current catalogue.

**Key trade-off**: APM is **pre-1.0** (v0.8.11, 6 April 2026), a community-driven project formally transferred to the `microsoft/` org in February 2026 — Microsoft customer support **does not back it** (`SUPPORT.md` declares this explicitly). Bus factor is roughly 2 in practice (Daniel Meppiel + Sergio Sisternes from EPAM). The risk of breaking changes before 1.0 is real. The alternatives (ARM, Lola, Claude Code native, Vercel skills, SkillKit, OpenSkills) **do not pass the must-haves**: either GPL-3.0 / GPL-2.0 licensing, no multi-target support, or no transitive dependencies + lockfile.

## 3. Detailed review of the top three

### 3.1 Microsoft APM — primary

**Formal parameters.** Repository `github.com/microsoft/apm`, docs `microsoft.github.io/apm`, licence **MIT**. Creator and lead maintainer — **Daniel Meppiel** (@danielmeppiel); co-author — Sergio Sisternes from EPAM (@sergio-sisternes-epam). The repo was transferred from `danielmeppiel/apm` to `microsoft/apm` on **15 February 2026** (Discussion #86); v0.7.3 added Microsoft OSS compliance files. Current version — **v0.8.11 (6 April 2026)**, 40 releases, ~889 commits; language — Python (95%). Stars sit in the 1.2k–1.7k range (active growth). Install: `pip install apm-cli`, `brew install microsoft/apm/apm`, Scoop for Windows, native binaries `aka.ms/apm-unix` / `aka.ms/apm-windows`. Support channel — GitHub Issues / Discussions only; `SUPPORT.md` states directly: 'support happens via GitHub Issues/Discussions, not through Microsoft customer support channels'.

**Functional review.** All eight must-haves are covered:

1. **Primitives (7, including every required one)**: instructions (`.instructions.md` with `applyTo` glob), skills (Anthropic `SKILL.md`), prompts (`.prompt.md` — slash commands), agents (`.agent.md`), hooks (`hooks/*.json`), plugins (`plugin.json`), MCP servers. Source: microsoft.github.io/apm/reference/primitive-types/.
2. **Targets** — native deployment into `.github/`, `.claude/`, `.cursor/`, `.opencode/`, `.agents/skills/` (Codex). Compilation of `AGENTS.md` / `CLAUDE.md` via `apm compile` for Gemini and others. **Key design principle**: 'APM does not invent new abstractions — it manages the ones that already exist' — generated files remain native; if APM is removed, they continue to work.
3. **Versioned dependencies** — ✅ partially. Pinning is supported via `#v1.0.0`, `#v2.0`, `#<branch>`, `#<commit-sha>`; **explicit semver range resolution (`^`, `~`) is not present in the current documentation** — refs are treated as exact git references. Transitive dependencies are tracked with a `depth: N` field in the lockfile. `apm.lock.yaml` pins every dependency to an exact commit SHA, records `deployed_files` (for safe cleanup), and `mcp_servers` (for detecting stale entries).
4. **Multi-repo sources** — ✅ fully: 'Any git host: GitHub, GitLab, Bitbucket, Azure DevOps, GitHub Enterprise, self-hosted — if you can `git clone` it, APM can install it. No central registry required.' Canonical normalisation: shorthand, HTTPS, SSH, FQDN — all reduced to a single form.
5. **Metadata** — manifest format is `apm.yml`; mandatory fields include `name`, `version`, `dependencies`. A JSON Schema is published in the docs.
6. **Marketplace** — CLI: `apm marketplace add/browse/search`; both the Copilot CLI and Claude Code marketplace formats are supported via `marketplace.json`. There is no dedicated web UI, but integration with the Copilot catalogue exists.
7. **Security (a unique strength)** — install-time scanner for hidden Unicode characters (tag chars U+E0001–U+E007F, bidi overrides U+202A–U+202E, SMP variation selectors — **blocks the install, exit 1**); `apm audit --strip` removes them; output in text / json / **sarif** / markdown. **MCP transitive-trust rule**: MCP servers at `depth=1` are auto-trusted; `depth>1` requires `--trust-transitive-mcp`. **Policy engine** (`apm audit --ci --policy`) — org-level rules, `apm-policy.yaml`, integration with GitHub Rulesets.
8. **CI / CD for self-hosted GitLab**: the official `microsoft/apm-action@v1` GitHub Action; for air-gapped infrastructure — `PROXY_REGISTRY_URL` / `PROXY_REGISTRY_TOKEN` / `PROXY_REGISTRY_ONLY=1` (Artifactory-style proxy); the proxy path is recorded in the lockfile.

**Real manifest example:**

```yaml
name: your-project
version: 1.0.0
dependencies:
  apm:
    - anthropics/skills/skills/frontend-design        # skill from a subdirectory
    - github/awesome-copilot/plugins/context-engineering
    - github/awesome-copilot/agents/api-architect.agent.md
    - microsoft/apm-sample-package#v1.0.0             # pin to tag
    - git: https://gitlab.acme.corp/platform/coding-standards.git
      path: instructions/security
      ref: v2.0
  mcp:
    - io.github.github/github-mcp-server              # reference to registry
```

**Pros.** The only tool in the OSS category with real transitive resolution + lockfile + policy engine. Real-world security mitigation — the Sourcegraph Amp incident is documented (invisible Unicode prompt injection, embracethered.com); a study of 50 popular `.cursorrules` files found 6 with hidden characters (issue #312). Air-gapped mode via a proxy registry. Active release cycle; star count rising.

**Cons.** Pre-1.0 — breaking changes are expected (the 0.8.x series added marketplace integration, `--global`, Codex support). Bus factor is 2 active maintainers. Installation depends on Python (offset by native binaries). No semver ranges. No cryptographic package signatures — only Unicode / policy sanity checks. The Microsoft brand here signals an OSS umbrella, not a supported product.

**When to choose.** When a single governance layer over Claude Code + Cursor + OpenCode + Codex is needed, reproducibility (`apm.lock.yaml`) is critical, an in-house CI infrastructure (GitLab) is in place, and a policy engine is required. Your case is an ideal fit.

### 3.2 Red Hat Lola — fallback for multi-target via AGENTS.md

**Formal parameters.** `github.com/RedHatProductSecurity/lola`, licence **GPL-2.0-or-later** ⚠️, current version **v0.4.0 (3 February 2026)**, last commit 26 March 2026. Stars — **13**, forks — 7, contributors — 6 (Igor Brandao, Katie Mulliken and others, one of them — claude-bot). Python 100%. Install: `uv tool install git+https://github.com/RedHatProductSecurity/lola`. The companion project `RHEcosystemAppEng/agentic-collections` (7 personas, 60+ skills for RHEL / OpenShift / Ansible) is separate, under Apache-2.0.

**Functional review.** Covers 3 / 8 must-haves fully: multi-target (Claude Code / Cursor / Gemini CLI / OpenCode via AGENTS.md references), multi-repo sources (any git URL), and a YAML-format marketplace convenient for migration from REGISTRY.yaml. Does not cover: transitive dependencies and lockfile are **absent entirely**; no signatures or sandboxing (only path-traversal validation for install hooks); no enterprise managed allowlist. MCP support is mentioned in the 'AI Context Module' description, but the mechanism for injection into `.mcp.json` is not detailed in the README. **Codex is not supported.**

**Manifest example (`marketplace.yml`):**

```yaml
name: My Marketplace
description: Curated collection of AI skills
version: 1.0.0
modules:
  - name: git-workflow
    description: Git workflow automation skills
    version: 1.0.0
    repository: https://github.com/user/git-workflow.git
    tags: [git, workflow]
    hooks:
      pre-install: scripts/check-python.sh
      post-install: scripts/verify.sh
  - name: monorepo-skills
    repository: https://gitlab.acme.corp/platform/monorepo.git
    path: packages/lola-skills
```

**Licensing risk.** GPL-2.0-or-later on the CLI — if Lola is used as a **stand-alone tool** (not embedded, not linked into proprietary code), copyleft does not 'infect' the product. But **legal review is mandatory** before embedding; a fork becomes a copyleft burden.

**When to choose.** If APM does not fit for some reason (for example, a pure Red Hat stack with OpenShift integration is required) and multi-target is more critical than supply-chain discipline. For your case — **not primary**, possibly a component of the hybrid for AGENTS.md-first ecosystems.

### 3.3 Claude Code native plugin marketplaces — extra layer for Claude-heavy teams

**Formal parameters.** Part of Anthropic Claude Code (proprietary CLI, but the plugin format is open and compatible with the marketplace manifest schema `https://anthropic.com/claude-code/marketplace.schema.json`). The plugin mechanism matured in 2026: a `strictKnownMarketplaces` allowlist appeared for managed enterprise settings, cache isolation (`~/.claude/plugins/cache`), path-traversal protection, `--keep-data` on uninstall. In the late-2025 / early-2026 CHANGELOG: 'Fixed plugin install not honoring dependencies declared in `plugin.json` when the marketplace entry omits them; `/plugin install` now lists auto-installed dependencies' — **a transitive resolver appeared, but is not publicly documented**; issue #9444 on shared 'common-core' libraries is still open.

**Functional review.** Primitives — full support (skills, MCP, commands, sub-agents, hooks, LSP, monitors, output styles). Targets — **Claude Code only**. Multi-repo — any git URL (`/plugin marketplace add https://gitlab.corp/plugins.git#v1.0.0`). Managed enterprise controls — the strongest in the entire sample: `strictKnownMarketplaces` allowlist, `extraKnownMarketplaces` for team provisioning, managed scope fixes the set of plugins, auto-update is OFF for third-party by default. **No lockfile. No signatures.** Anthropic states directly: 'Anthropic does not control what MCP servers, files, or other software are included in plugins and cannot verify that they work as intended.'

**`.claude-plugin/marketplace.json` example:**

```json
{
  "$schema": "https://anthropic.com/claude-code/marketplace.schema.json",
  "name": "acme-internal-marketplace",
  "version": "1.0.0",
  "owner": { "name": "Acme Platform", "email": "platform@acme.corp" },
  "plugins": [
    {
      "name": "acme-devops-tools",
      "source": { "source": "url", "url": "https://gitlab.acme.corp/ai/devops-plugin.git" },
      "version": "2.1.0",
      "strict": true
    }
  ]
}
```

**When to choose.** For corporate teams using **Claude Code only**, with strict governance via managed settings — the native Anthropic stack. In your multi-target case — **not primary**, but APM generates this format natively and integrates with it, so it is used as a deployment target.

## 5. Hybrid architectures

Since no OSS tool covers **everything** ideally, a hybrid of 3–4 components with APM as the orchestration layer is recommended.

**Recommended architecture for your context:**

|Layer|Tool|Role|
|---|---|---|
|Orchestration + transitive resolution + lockfile|**Microsoft APM**|Source of truth, `apm.yml` + `apm.lock.yaml`, deploy into every target|
|Internal package registry|**Self-hosted GitLab** + **LiteLLM marketplace pattern** (optional Admin UI)|Private git-backed source; `PROXY_REGISTRY_URL` for air-gapped|
|MCP-specific catalogue|**MCPM** (mcpm.sh) **+ MCP Registry** (community subregistry)|Additional source of MCP servers with a mature community catalogue; private subregistry as a mirror|
|Supply chain scanning|**Snyk Agent Scan** (ex Invariant mcp-scan) + **APM Unicode sanitiser**|Pre-install CI gate: prompt injection, malware payloads, hidden unicode, tool poisoning|
|Runtime observability|Claude Code `OpenTelemetry` + Prometheus + VictoriaMetrics|Audit trail for agent invocations (gap in the MCP 2026 standard)|
|Policy enforcement|`apm audit --ci --policy apm-policy.yaml` + GitLab CI required job|Org-level allowlist / denylist, MCP transport controls, manifest requirements|

**Data flows.** `apm.yml` in the repository → GitLab CI runs `apm audit --ci --policy` + `snyk-agent-scan` → on success `apm install` on the developer's machine or inside a container → deploy into `.claude/`, `.cursor/`, `.opencode/`. MCP servers come either through `apm.yml` (for infra-managed cases) or through MCPM for ad-hoc discovery. `apm.lock.yaml` is committed to git for reproducibility.

**Alternative minimal hybrid** (if MCPM / Snyk are not desired immediately): APM + built-in `apm audit` + proxying through Artifactory → the result is a self-contained stack with supply-chain at the level of hidden-Unicode + policy engine.

## 6. Recommendation and adaptation plan

**Choice: Microsoft APM as primary, with gradual integration of Snyk Agent Scan into CI and MCPM as a complementary catalogue afterwards.**

**Migration plan from `REGISTRY.yaml` + Python resolver:**

1. **Weeks 1–2 — pilot.** On a single test repository: install `apm-cli`, port 3–5 skills from the current REGISTRY.yaml into the `apm.yml` format. Verify `apm install` in Claude Code / Cursor / OpenCode locally. Compare behaviour with the current Python resolver on transitive dependencies.
2. **Weeks 3–4 — registry setup.** Raise an internal GitLab group as the primary source (`apm` dependencies → `git: https://gitlab.acme.corp/...` entries). Configure `PROXY_REGISTRY_URL` via Artifactory for air-gapped nodes. Generate `apm-policy.yaml` with an org-level allowlist: permitted sources only `github.com/anthropics/`, `github.com/microsoft/`, `gitlab.acme.corp/*`.
3. **Weeks 5–6 — CI integration.** `microsoft/apm-action@v1` (if GitHub is used) or its equivalent in GitLab CI: `apm audit --ci --policy --output sarif` as a required status check. Add Snyk Agent Scan as a second gate (`uvx snyk-agent-scan@latest`).
4. **Weeks 7–8 — migration at scale.** Auto-generate `apm.yml` from the current REGISTRY.yaml with a script (YAML → YAML is a straightforward transformation). All 300+ components migrated, each with `apm.lock.yaml` committed to git.
5. **Month 3+** — rollout across all teams via managed `.claude/settings.json` with `strictKnownMarketplaces`.

**APM gaps and compensation:**

|What APM does not cover|How to compensate|
|---|---|
|Semver ranges (`^`, `~`) — only exact refs|Use **tag-based pinning** (`#v1.2.0`) + a regular `apm update` workflow with review. If needed, write a wrapper script that translates range → latest-matching-tag before `apm install`|
|Cryptographic package signatures|Layer Snyk Agent Scan + Sigstore / Cosign signing of internal skills on the GitLab CI side; validate via a custom hook in `apm audit`|
|MCP server registry discovery|MCPM or a direct reference via the MCP Registry (the `io.github.owner/name` notation is already supported by APM in the `mcp:` section)|
|Web UI for marketplace|LiteLLM marketplace Admin UI as a self-hosted front end; the Claude Code native `/plugin` interactive UI for end users|
|Runtime prompt injection|Invariant Guardrails (commercial) or custom hooks with Parry (community) on tool I/O|
|No Microsoft support|Actively participate in GitHub Discussions; consider funding via GitHub Sponsors for bus-factor mitigation|

**Risks over a 12–24 month horizon.**

1. **APM pre-1.0 → 1.0 breaking changes** — a realistic threat. Mitigation: track the CHANGELOG, pin v0.8.x for production via a proxy registry, and regularly test the upgrade path on staging.
2. **Competition from `gh skill`** — GitHub announced the official extension on 16 April 2026; it uses the agentskills.io spec and provenance pinning. Over a 12-month horizon it may become the **de facto standard** for skill distribution thanks to GitHub distribution and CI integration. Mitigation: monitor; APM is already compatible with the agentskills.io spec, so skills are portable back if a pivot becomes necessary.
3. **Anthropic native plugin marketplace stabilises** — the transitive resolver gets documented, a lockfile arrives, target support expands (unlikely, but observe). Mitigation: APM natively deploys into the `.claude-plugin/` format, so bilingualism will be organic.
4. **AGENTS.md + SKILL.md standards under the Linux Foundation (AAIF, since 9 December 2025)** — consolidation will continue and fragmentation will fall. This is **good for APM** as a vendor-neutral manager.
5. **Supply-chain incidents will grow** — as of April 2026 the registered list already includes CVE-2025-6514 (mcp-remote RCE, CVSS 9.6), CVE-2025-59528 (Flowise RCE, CVSS 10.0), CVE-2025-59536 / CVE-2026-21852 (Claude Code config-driven RCE, fixed in 2.0.65+), 'ClawJacked' CVE-2026-28363 (OpenClaw DNS rebinding), the ToxicSkills campaign (Snyk reports **36.82% of uploaded skills with security flaws, 1467 malicious payloads**), and 'Claudy Day' data exfiltration in default Claude.ai sessions. A supply-chain layer (Snyk Agent Scan + Unicode sanitiser + policy engine) **is not optional**.
6. **APM bus-factor risk** — 2 active maintainers for a project carrying Microsoft branding. Possible scenarios: (a) team expansion via Microsoft, (b) a community fork if maintainers leave, (c) a merger with `gh skill` during standard consolidation. Mitigation: internal competence to read APM code (Python, ~5K LOC); a ready fallback to Lola as a stable vendor-neutral multi-target manager.

## Conclusion

**The main takeaway that separates this analysis from a shallow star-count comparison:** as of April 2026 **Microsoft APM is the only OSS candidate with transitive dependencies and a deterministic lockfile**, and requirement #3 turns that into a blocker. The other, more popular tools (Vercel skills 7.5k ⭐, OpenSkills 8.8k ⭐) fail on this requirement — they either symlink or pull-latest from the main branch, which is incompatible with supply-chain discipline for 300+ components.

**A second counter-intuitive finding**: Lola's GPL licence is not a theoretical risk but a real blocker for a corporate adopter that plans embedding or custom extensions.
